# projectH Pipeline Runtime 문서 세트 안내
버전: v1.1  
작성일: 2026-04-10  
대상: `pipeline-launcher.py` / `start-pipeline.sh` / `watcher_core.py` 중심의 internal runtime 정비

## 1. 문서 목적

이 문서 세트는 `pipeline launcher` 안정성 저하 문제를 해결하기 위한 개발·설계·검증 문서를 한 번에 정리한 개정판입니다.  
핵심 목표는 다음 두 가지입니다.

1. `tmux + watcher + 파일 스캔 + pane 출력 해석` 중심의 **상태 추론 구조**를 줄이고,
2. **supervisor 단일 진실원천(single source of truth)** 구조로 전환하여 24시간 무인 운용 안정성을 확보하는 것입니다.

## 2. 이번 개정판에서 고정한 핵심 변경점

### 변경 1. compatibility 범위를 좁혔습니다.
기존 `.pipeline/*.md` 전체를 export-only로 보는 표현을 제거했습니다.  
**export-only 대상은 legacy status surface만** 허용하며,  
`.pipeline/claude_handoff.md`, `.pipeline/gemini_request.md`, `.pipeline/gemini_advice.md`, `.pipeline/operator_request.md` 같은 **rolling control slot은 제외**합니다.

### 변경 2. status v1에 control 블록을 추가했습니다.
현재 운영 화면과 handoff 흐름이 유지되도록 `active_control_file`, `active_control_seq`, `active_control_status`, `active_control_updated_at`를 status 필수 필드로 명시했습니다.

### 변경 3. 마이그레이션 초반 writer ownership를 고정했습니다.
최종 목표는 supervisor 단일 writer이지만, 전환 초반에는 **watcher-exporter 1개만 임시 writer**로 허용하고, supervisor는 shadow mode를 거친 뒤 takeover하도록 정리했습니다.

### 변경 4. wrapper/sentinel event를 open question에서 필수 계약으로 승격했습니다.
wrapper 위치, 필수 이벤트(`READY`, `HEARTBEAT`, `DISPATCH_SEEN`, `TASK_ACCEPTED`, `TASK_DONE`, `BROKEN`), 수집 경로, 임시 parser 허용 범위를 설계 문서에 명시했습니다.

### 변경 5. receipt의 의미를 좁혀 충돌을 줄였습니다.
receipt는 **runtime close authority**이고, `/work`와 `/verify`는 여전히 **persistent truth / audit artifact**로 남는다고 명시했습니다.

### 변경 6. QA 게이트 표현을 정리했습니다.
`release gate` 대신 **internal tooling adoption gate**라는 표현으로 정리해 현재 repo 범위와 맞췄습니다.

### 변경 7. 별도 마이그레이션 체크리스트를 추가했습니다.
RUNBOOK은 목표 구조 기준 운영 문서이고, cutover 절차·shim 제거 순서·rollback decision point는 별도 문서로 분리했습니다.

### 변경 8. status truth와 mirror 경계를 고정했습니다.
authority는 run-scoped `status.json` 1개와 append-only `events.jsonl` 1개로 두고,  
`.pipeline/state/turn_state.json`는 migration 동안만 export-only mirror로 둡니다.

### 변경 9. UI boundary를 고정했습니다.
브라우저 controller UI는 controller HTTP API만 읽고,  
`pipeline_gui`와 `pipeline-launcher`는 로컬 thin client로서 shared runtime helper/CLI만 읽는 것으로 정리했습니다.

### 변경 10. receipt authority를 고정했습니다.
round-close receipt는 supervisor만 생성하며 `/verify`를 대체하지 않는다고 명시했습니다.

## 4. 기본 가정

아래는 사실이 아니라 본 문서 작성 시 둔 가정입니다.

- 운영자는 여전히 `attach` 또는 live debug 경로를 원합니다.
- 당장 모든 lane을 PTY 기반으로 갈아엎기보다, `tmux backend`를 먼저 adapter 뒤로 숨기는 단계적 전환이 더 현실적입니다.
- `pipeline runtime`은 당분간 release-candidate browser contract와 분리된 internal/operator tooling으로 유지됩니다.
- `controller.server`를 바로 runtime 본체로 키우기보다, 공용 runtime supervisor 레이어를 먼저 만드는 편이 안전합니다.

## 7. 한 줄 결론

이번 정비의 핵심은 `tmux를 즉시 제거하는 것`이 아니라,  
**launcher에서 상태 추론을 걷어내고, control/receipt/wrapper 계약을 명시한 supervisor 중심 구조로 전환하는 것**입니다.


## 이번 개정에서 보강한 점
- Phase 3 완료 기준을 status/control current truth writer 범위로 좁혔습니다.
- receipt writer ownership은 Phase 5와 Stage E에서 확정되도록 정렬했습니다.
- ADR-007에서 export-only compatibility 대상을 legacy status surface로 한정하고 rolling control slot 제외를 명시했습니다.
- status truth를 `run-scoped status.json + events.jsonl`로 고정하고 `turn_state.json`를 mirror로 내렸습니다.
- 브라우저 controller UI는 controller HTTP 전용으로 고정하고, 로컬 thin client는 runtime helper/CLI 경계로 고정했습니다.
- wrapper 필수 이벤트를 5개로 줄이고 lane별 recovery 정책을 명시했습니다.
