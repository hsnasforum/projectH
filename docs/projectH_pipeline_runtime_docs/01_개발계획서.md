# projectH Pipeline Runtime 안정화 개발계획서
버전: v1.1  
작성일: 2026-04-10  
문서 성격: 개발 계획서 / 실행 초안

## 1. 배경

현재 pipeline launcher 계층은 실행기이면서 동시에 상태 관측기 역할을 같이 수행합니다.  
이 구조에서는 다음이 동시에 일어납니다.

- tmux 세션 존재 여부 확인
- watcher pid 파일 확인
- 최신 `work` / `verify` 산출물 스캔
- `watcher.log` tail 해석
- 각 lane의 pane 출력 문자열 해석
- control slot 선택 결과를 여러 곳에서 다시 추론

이 방식은 화면에서는 편리하지만, 24시간 안정성 기준에서는 약합니다.  
같은 시스템 상태를 서로 다른 힌트로 재구성하기 때문에 split-brain, false-ready, stale-state, orphan-process, duplicate-dispatch, 복구 불일치가 반복될 가능성이 높습니다.

## 2. 문제 정의

### 2.1 현재 핵심 문제
1. launcher가 너무 많은 진실원천을 직접 읽습니다.
2. start/stop와 상태 관측이 강하게 결합되어 있습니다.
3. lane 상태 판정이 vendor CLI의 UI 문자열에 의존합니다.
4. control plane과 status surface의 경계가 명확하지 않습니다.
5. 전역 `.pipeline/state`, `.pipeline/locks`, `.pipeline/manifests` 초기화는 재시작 복구와 장시간 운용에 불리합니다.
6. 운영성과 디버깅 편의 때문에 추가된 보정 로직이 누적되면서 시스템이 무거워지고 예측 가능성이 떨어집니다.

### 2.2 해결 대상
이번 프로젝트는 아래를 해결 대상으로 삼습니다.

- launcher 상태 추론 제거
- supervisor 단일 상태 작성자 도입
- status v1 + control block 고정
- tmux runtime 직접 의존 축소
- run-scoped 상태 구조 도입
- receipt 기반 round close 판정 도입
- wrapper event 기반 readiness 정의
- 24시간 soak test 통과

### 2.3 이번 단계에서 해결하지 않는 것
- 문서 비서 본체(`app.web`)의 기능 확장
- durable user memory, broader structured correction memory
- 모델 성능 개선
- 즉시 전체 lane PTY 전환
- 장기적으로 가능한 Windows-native runtime 구현

### 2.4 "24시간 soak"의 위치
이 계획서에서 반복 등장하는 "24시간 soak"과 "24h soak gate"는 **이번 launcher 안정화 프로젝트의 최종 채택(adoption) / cutover baseline gate**를 뜻합니다.
채택 이후의 **day-to-day 기본 검증**은 `05_운영_RUNBOOK.md §3.5 현재 검증 원칙`이 authority이며,
`launcher live stability gate + incident replay + 실제 작업 세션` 세 축으로 운영됩니다.
따라서 이 계획서의 24h/6h soak 표현은 historical baseline 및 adoption/cutover 전용으로 읽고, 기본 검증으로 다시 끌어올리지 않습니다.

### 2.5 자동화 완성 목표
자동화의 완성 목표는 ordinary next-step, ambiguity, stall, session rollover, recovery 상황에서 사용자를 호출하지 않는 것입니다.
문제가 발생하면 implement / verify-handoff / advisory owner가 `/work`, `/verify`, current docs, runtime evidence를 기준으로 먼저 회의하고 하나의 다음 control로 좁힙니다.

재귀학습과 진화적 탐색은 현재 단계에서 모델 가중치 학습이 아니라 repo-local operational learning입니다.
반복 문제는 named incident, focused replay, owning boundary, shared helper, truthful runtime surface로 남기고,
후보 탐색은 current evidence와 milestone에 묶인 bounded comparison으로 닫습니다.
삭제/덮어쓰기, auth/credential, approval-record, truth-sync, publication boundary 같은 실제 위험은 여전히 명시적이고 감사 가능한 승인 경계로 유지합니다.

## 3. 추진 원칙

### 원칙 1. launcher는 thin client여야 합니다.
launcher는 상태를 계산하지 않고 command 전송과 status 조회만 수행합니다.

### 원칙 2. 최종 단일 writer는 supervisor입니다.
최종 상태에서는 status, event, receipt, run lifecycle을 supervisor만 씁니다.  
단, 전환 초반에는 **watcher-exporter 1개만 임시 writer**로 허용하고, supervisor는 shadow mode를 거쳐 takeover합니다.
authority 경로는 run-scoped `status.json`과 `events.jsonl`이며, `.pipeline/state/turn_state.json`는 migration 동안 export-only mirror로만 유지합니다.

### 원칙 3. tmux는 당분간 backend adapter로 남깁니다.
tmux attach/debug의 운영 효용은 유지하되, 상위 계층에서 tmux 세부를 직접 읽지 않게 만듭니다.

### 원칙 4. artifact와 state를 분리합니다.
`/work`, `/verify`는 persistent truth / audit artifact입니다.  
round 완료 여부는 receipt가 판단하며, round-close receipt 생성 권한은 supervisor만 가집니다.

### 원칙 5. control plane과 status surface를 분리합니다.
legacy status surface는 export-only로 둘 수 있지만,  
rolling control slot은 전환 기간에도 canonical control plane으로 유지합니다.

### 원칙 6. 전환은 단계적으로 합니다.
한 번에 PTY로 가는 big bang migration 대신,  
`status schema → shadow mode → supervisor takeover → TmuxAdapter → run-scoped state → receipt → wrapper → optional PTY pilot` 순서로 전환합니다.

### 원칙 7. 브라우저 UI와 로컬 thin client 경계를 분리합니다.
브라우저 controller UI는 controller HTTP만 읽습니다.  
`pipeline_gui`와 `pipeline-launcher`는 로컬 thin client로 유지하되 shared runtime helper/CLI만 사용하고, repo 파일 직접 스캔이나 pane/log current truth 추론으로 돌아가지 않습니다.

### 원칙 8. 하드코딩, 중복, 책임 집중을 금지합니다.
runtime logic은 current branch, commit SHA, `CONTROL_SEQ`, pane id, 한국어 label, 특정 operator prose, one-off control body를 하드코딩하지 않습니다.
watcher / supervisor / launcher / controller가 같은 truth를 near-copy로 재해석하지 않고, owning module이나 shared helper를 사용합니다.
한 파일이나 한 함수에 unrelated branch를 계속 쌓아 유지보수성을 낮추지 않으며, parsing / labeling / control writing / lane surface / event contract 책임이 커지면 명확한 owner로 추출합니다.

채택 이후의 상시 검증은 24h soak 재실행이 아니라 `05_운영_RUNBOOK.md §3.5`의 3축(launcher live stability + incident replay + 실제 작업 세션)을 기본으로 합니다.

|---|---:|
| 24h soak 동안 manual intervention | 0회 |
| duplicate dispatch | 0건 |

## 5. 단계별 개발 계획

## Phase 0. 현행 안정화 계측
목표: 현재 구조의 실패 모드를 측정 가능하게 만듭니다.

### 작업
- launcher/watcher/start/stop 경로에 공통 `run_id` 주입
- 현재 실패 유형 taxonomy 정의
- start/stop/restart/dispatch/verify/recover 이벤트를 JSONL로 남기기
- 24시간 soak test용 synthetic workload 준비
- 장애 재현 스크립트 준비

### 산출물
- 이벤트 taxonomy
- soak harness
- 장애 재현 체크리스트
- baseline metrics

### 완료 기준
- 최소 24시간 baseline 실행 결과를 수치로 남길 수 있어야 합니다.
- “불안정하다”를 로그와 지표로 설명할 수 있어야 합니다.

## Phase 1. status schema v1 고정 + 임시 writer 명문화
목표: 현재 상태 surface를 먼저 단일 schema로 고정합니다.

### 작업
- `status.json` v1 정의
- `control` 블록 정의
- watcher-exporter 1개만 임시 status writer로 허용
- launcher/GUI가 새 status만 읽도록 전환
- direct tmux/log/file scan UI를 감추거나 비활성화

### 산출물
- status schema v1
- control block 정의
- 임시 ownership 규칙 문서
- launcher status reader 전환

### 완료 기준
- production `status.json` 작성자가 딱 1개여야 합니다.
- launcher가 `capture-pane`, `watcher.log`, `rglob("*.md")`를 직접 호출하지 않아야 합니다.
- `active_control_file`, `active_control_seq`가 status로 노출되어야 합니다.

## Phase 2. supervisor shadow mode
목표: supervisor를 도입하되, 초기에는 shadow 비교만 수행합니다.

### 작업
- `pipeline_runtime.supervisor` 도입
- 동일 run에 대해 shadow status 계산
- shadow output과 production status diff 수집
- start/stop/restart/attach command channel 정의

### 산출물
- supervisor skeleton
- shadow diff log
- launcher-supervisor command API

### 완료 기준
- supervisor shadow output이 production status와 핵심 필드에서 일관돼야 합니다.
- takeover 전까지 production `status.json` 작성자는 계속 watcher-exporter 1개여야 합니다.

## Phase 3. supervisor takeover + Launcher thin client
목표: supervisor를 status/control current truth writer로 승격합니다.

### 작업
- production status writer를 supervisor로 전환
- watcher-exporter 비활성화
- launcher 명령 경로를 supervisor로 고정
- direct control/status 계산 경로 제거

### 산출물
- supervisor production mode
- watcher-exporter off 플래그
- thin launcher

### 완료 기준
- production `status.json`과 `control` block current truth writer가 supervisor 한 곳이어야 합니다.
- watcher/GUI/launcher가 각자 상태나 control을 따로 계산하지 않아야 합니다.
- receipt writer ownership은 Phase 5 범위이며, Phase 3 완료 기준에 포함하지 않습니다.

## Phase 4. TmuxAdapter 분리
목표: tmux를 runtime substrate가 아니라 adapter로 격하합니다.

### 작업
- `TmuxAdapter` 인터페이스 정의
- lane spawn / terminate / send_input / capture_tail / attach / health 분리
- start/stop shell script 의존 최소화
- tmux session metadata를 backend 전용 파일로 분리

### 산출물
- adapter 추상화
- tmux backend 구현
- backend health check

### 완료 기준
- supervisor 외 계층에서 tmux 명령 문자열을 직접 사용하지 않아야 합니다.

## Phase 5. Run-scoped state + receipt
목표: stale state와 restart ambiguity를 줄입니다.

### 작업
- `.pipeline/runs/<run_id>/...` 구조 도입
- `current_run.json` 도입
- receipt schema 도입
- receipt current truth writer를 supervisor로 확정
- round close를 artifact가 아니라 receipt 기준으로 판정
- state/locks/manifests 전역 초기화 제거

### 산출물
- run directory schema
- receipt writer
- restart/recovery 규칙

### 완료 기준
- receipt current truth writer가 supervisor 1개여야 합니다.
- 이전 run의 state, lock, manifest, pid가 현재 run처럼 보이면 안 됩니다.
- receipt가 없으면 round 종료가 확정되지 않아야 합니다.
- `/work`, `/verify`가 receipt로 대체되면 안 됩니다.

## Phase 6. Wrapper contract + fault-tolerance
목표: readiness와 복구를 문자열 추론이 아니라 공식 이벤트로 옮깁니다.

### 작업
- lane wrapper 위치 고정
- 필수 wrapper event 정의: `READY`, `HEARTBEAT`, `DISPATCH_SEEN`, `TASK_ACCEPTED`, `TASK_DONE`, `BROKEN`
- 임시 parser 허용 범위 제한
- lane별 recovery 정책 고정:
  implement owner는 post-accept blind replay 금지
  verify/advisory owner는 bounded auto-retry 허용
- orphan lease 정리
- lane crash 자동 재기동
- degraded mode / quarantine / restart recovery

### 산출물
- wrapper event 채널
- recovery state machine
- degraded mode 정책
- fault injection 결과 보고서

### 완료 기준
- STARTING/READY 판정이 pane 문자열이 아니라 wrapper 이벤트 기준이어야 합니다.
- 강제 종료, pane dead, state corruption, stale pid, manifest mismatch를 수동 개입 없이 안전정지 또는 자동복구해야 합니다.

## Phase 7. Internal tooling adoption gate
목표: runtime 채택 여부를 결정합니다. 이 Phase의 24시간 soak은 **채택/baseline gate 1회 수행**이며, 이후 day-to-day 기본 검증을 대체하지 않습니다.

### 작업
- integration / e2e / fault injection 재검증
- 24시간 soak (채택 baseline 기준)
- runbook / migration checklist 동기화
- defect triage 및 잔여 위험 판단

### 산출물
- QA 결과 보고서
- soak log bundle
- adoption sign-off

### 완료 기준
- 24시간 soak pass (채택 baseline)
- duplicate dispatch 0
- receipt 누락 0
- control/status 불일치 0
- 채택 이후 상시 검증은 `05_운영_RUNBOOK.md §3.5` 3축으로 이관됨을 명시

## Phase 8. 선택 과제: PTY pilot
목표: tmux 제거 가능성을 검증하되, 전체 교체는 하지 않습니다.

### 작업
- `PtyAdapter` 실험 구현
- verify lane 또는 advisory lane 1개만 pilot
- TmuxAdapter와 기능 parity 비교
- 운영성/성능/복구성 비교

### 산출물
- PTY pilot 비교 보고서
- 도입/보류 결정

### 완료 기준
- PTY가 실제로 더 안정적이라는 증거가 있을 때만 확대합니다.

## 9. 리스크와 대응

### 리스크 1. 기존 보정 로직 제거 과정에서 가시성 저하
대응:
- status schema와 events.jsonl를 먼저 고정
- shadow mode에서 차이를 먼저 기록
- 기존 log tail은 supervisor 내부 compatibility mode로만 한시 유지

### 리스크 2. 전환 초반 single-writer 원칙 붕괴
대응:
- production status writer는 항상 1개만 허용
- shadow output은 별도 파일만 사용
- takeover 시 watcher-exporter off를 같은 변경 세트에서 수행

### 리스크 3. compatibility 범위가 다시 넓어짐
대응:
- export-only는 legacy status surface로만 제한
- rolling control slot은 명시적으로 제외
- control block은 status에 노출하되 control slot 자체를 compat로 복제하지 않음

### 리스크 4. wrapper 구현 난이도
대응:
- wrapper는 최소 이벤트부터 구현
- parser가 필요하면 wrapper 내부에만 가둠
- raw vendor text를 launcher/GUI까지 노출하지 않음

### 리스크 5. receipt 설계가 늦어져 상태 판정이 다시 artifact 의존으로 회귀
대응:
- receipt schema를 Phase 1 말에 먼저 고정
- QA 게이트에 “receipt 누락 시 round close 실패” 조건 포함
- `/work`, `/verify`가 여전히 persistent truth임을 명시

## 10. 승인 게이트

다음 조건을 모두 만족해야 본 전환을 완료로 봅니다.

1. launcher가 tmux/log/work/verify를 직접 읽지 않습니다.
2. supervisor가 단일 상태 작성자입니다.
3. status v1이 control block을 포함합니다.
4. run-scoped state가 적용되었습니다.
5. round 종료는 receipt로만 확정됩니다.
6. readiness는 wrapper event 기준입니다.
7. 24시간 soak test를 1회 채택 baseline gate로 통과합니다. 채택 이후 상시 검증은 `05_운영_RUNBOOK.md §3.5` 3축을 기본으로 합니다.
8. `07_마이그레이션_체크리스트`와 `05_운영_RUNBOOK`이 실제 절차와 일치합니다.
