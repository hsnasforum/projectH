# projectH Pipeline Runtime 요구사항 명세서
버전: v1.1  
작성일: 2026-04-10  
문서 성격: 기능/비기능 요구사항 명세

## 1. 범위

본 문서는 `pipeline-launcher` 계층과 관련 runtime의 안정성 개선 요구사항을 정의합니다.

### 포함 범위
- launcher 명령 흐름
- supervisor 상태 관리
- tmux backend adapter
- watcher/orchestrator와 supervisor 경계
- run state, receipt, recovery, health
- status/control surface
- QA 및 운영에서 필요한 관측 정보
- migration ownership 규칙

### 제외 범위
- `app.web` shipped browser contract 기능
- reviewed-memory 기능 확장
- 장기 user memory
- 모델 품질 자체의 개선
- 전체 lane의 즉시 PTY 전환

## 2. 이해관계자

| 구분 | 관심사 |
|---|---|
| 운영자 | start/stop/restart/attach가 예측 가능해야 함 |
| 개발자 | 상태 모델과 backend 경계가 명확해야 함 |
| QA | 채택 baseline에 한해 24h soak와 fault injection 결과를 재현 가능하게 수집해야 함 (day-to-day 기본 검증은 `05_운영_RUNBOOK §3.5`) |
| 릴리즈 관리자 | internal tooling 변경이 shipped browser contract에 영향 주지 않아야 함 |

## 3. 기능 요구사항

## FR-001. Supervisor 단일 상태 작성자
시스템은 pipeline runtime의 current truth를 최종적으로 supervisor 한 곳에서만 기록해야 합니다.

**수용 기준**
- steady state에서 status/event/receipt current truth writer는 supervisor만 존재합니다.
- authority path는 run-scoped `status.json` 1개와 append-only `events.jsonl` 1개입니다.
- `.pipeline/state/turn_state.json`는 export-only mirror이며 current truth가 아닙니다.
- launcher는 상태 파일/API를 읽기만 합니다.
- watcher, GUI, launcher가 각자 현재 상태를 따로 계산하지 않습니다.

## FR-002. Launcher thin client화
launcher는 명령 전송과 상태 조회만 수행해야 합니다.

**포함 명령**
- start
- stop
- restart
- attach
- refresh / status 조회

**수용 기준**
- launcher는 tmux pane, watcher.log, `/work`, `/verify`를 직접 스캔하지 않습니다.
- launcher는 vendor CLI의 raw UI 문자열로 READY/WORKING을 판단하지 않습니다.

## FR-003. Backend adapter 추상화
runtime backend는 adapter 인터페이스 뒤에서 동작해야 합니다.

**최소 인터페이스**
- `spawn_lane`
- `terminate_lane`
- `restart_lane`
- `send_input`
- `capture_tail`
- `attach`
- `health`

**수용 기준**
- tmux backend 구현은 존재할 수 있으나, 상위 계층은 tmux 명령 상세를 몰라야 합니다.

## FR-004. Run-scoped 상태 구조
시스템은 각 실행을 고유 `run_id` 아래에서 관리해야 합니다.

**수용 기준**
- 현재 run과 이전 run의 상태 파일이 분리됩니다.
- 전역 state/lock/manifests 비우기로 현재 run을 시작하지 않습니다.
- `current_run.json` 또는 동등한 포인터가 존재합니다.

## FR-005. Receipt 기반 round close
round 완료는 artifact 생성이 아니라 receipt 기록으로 확정해야 합니다.

**필수 필드**
- run_id
- job_id
- round
- artifact_path
- artifact_hash
- target_lane
- verify_result
- closed_at

**수용 기준**
- receipt가 없으면 round는 `CLOSED`가 될 수 없습니다.
- receipt는 runtime close authority일 뿐, `/work`와 `/verify`를 대체하지 않습니다.
- `/work`와 `/verify`는 persistent truth / audit artifact로 유지됩니다.
- round-close receipt current truth writer는 supervisor 1개뿐입니다.

## FR-006. Recovery 지원
시스템은 lane crash, watcher crash, stale pid, stale lease, corrupt state를 탐지하고 복구 또는 안전정지해야 합니다.

**수용 기준**
- 각 실패 모드는 event에 기록됩니다.
- 복구 실패 시 상태는 `DEGRADED` 또는 `BROKEN`으로 명시됩니다.
- false READY나 silent failure는 허용되지 않습니다.
- implement owner lane은 `TASK_ACCEPTED` 이후 blind replay를 금지합니다.
- verify owner lane과 advisory owner lane은 completion fact 부재 시 bounded auto-retry를 허용합니다.

## FR-007. Observable events
시스템은 최소한 다음 이벤트를 순서 보장 가능한 형태로 남겨야 합니다.

- runtime_started
- runtime_stopped
- lane_spawned
- lane_ready
- lane_working
- lane_broken
- job_dispatched
- dispatch_suppressed
- verify_started
- verify_completed
- receipt_written
- recovery_started
- recovery_completed
- degraded_entered
- degraded_cleared

**수용 기준**
- JSONL 또는 동등한 append-only event log가 존재합니다.

## FR-008. Compatibility 모드
이행 기간 동안 기존 tooling과의 호환 경로가 있어야 합니다.

**수용 기준**
- export-only compatibility 대상은 **legacy status surface**로 제한합니다.
- `latest-work`, `latest-verify`, legacy status snapshot 같은 read model 보조 surface는 export-only로 제공할 수 있습니다.
- `.pipeline/implement_handoff.md`, `.pipeline/advisory_request.md`, `.pipeline/advisory_advice.md`, `.pipeline/operator_request.md` 같은 rolling control slot은 export-only 대상이 아닙니다.
- rolling control slot은 전환 기간에도 canonical control plane으로 유지됩니다.
- historical aliases `.pipeline/claude_handoff.md`, `.pipeline/gemini_request.md`, `.pipeline/gemini_advice.md`는 같은 logical slot의 read-only compatibility input으로만 허용하며, 별도 control plane이나 export-only read model로 취급하지 않습니다.
- legacy surface가 state authority가 되어서는 안 됩니다.

## FR-009. Health/status 조회
운영자는 현재 상태를 단일 조회 경로로 볼 수 있어야 합니다.

**최소 조회 정보**
- runtime_state
- current_run_id
- last_heartbeat_at
- lane별 상태
- active round
- 마지막 receipt
- degraded reason
- control block

**control block 최소 필드**
- active_control_file
- active_control_seq
- active_control_status
- active_control_updated_at

**수용 기준**
- 브라우저 controller UI는 repo 파일을 직접 읽지 않고 controller HTTP API만 소비합니다.
- `pipeline_gui`와 `pipeline-launcher`는 로컬 thin client로서 shared runtime helper/CLI를 사용할 수 있습니다.
- 어떤 UI/client도 pane, watcher.log, 최신 `/work`·`/verify`를 current truth readiness 판단 근거로 쓰지 않습니다.
- controller는 `/api/runtime/status`와 `/api/runtime/start|stop|restart` 또는 동등한 HTTP 경로를 제공합니다.
- controller status payload는 active profile의 `role_owners`/`prompt_owners` 같은 read-model metadata를 함께 제공할 수 있으며, browser는 이를 Office View label/home-zone 표시용으로만 사용합니다.

## FR-010. Attach/debug 지원
운영자는 필요 시 lane 또는 backend에 attach할 수 있어야 합니다.

**수용 기준**
- attach 경로는 supervisor가 중재합니다.
- attach 가능 여부가 status에 표시됩니다.

## FR-011. Migration 기간의 writer ownership
전환 중에도 production status writer는 한 번에 하나만 허용해야 합니다.

**수용 기준**
- supervisor takeover 전에는 watcher-exporter 1개만 production status writer로 허용합니다.
- supervisor shadow output은 별도 파일 또는 별도 채널로만 기록합니다.
- supervisor takeover 시 watcher-exporter는 같은 변경 세트에서 비활성화됩니다.
- 어느 단계에서도 서로 다른 두 writer가 같은 production `status.json`을 동시에 쓰면 안 됩니다.

## FR-012. Lane wrapper / sentinel event 계약
lane readiness와 work state는 공식 이벤트로 표면화되어야 합니다.

**필수 이벤트**
- `READY`
- `HEARTBEAT`
- `DISPATCH_SEEN`
- `TASK_ACCEPTED`
- `TASK_DONE`
- `BROKEN`

**수용 기준**
- wrapper는 vendor CLI의 immediate parent 또는 동등한 제어 경계로 배치됩니다.
- vendor UI 문자열 파싱이 필요하더라도 wrapper 내부에서만 수행합니다.
- launcher/GUI는 raw vendor text가 아니라 wrapper event 결과만 소비합니다.
- `DISPATCH_SEEN`은 최소 `job_id`, `control_seq`, `dispatch_id`를 포함해야 합니다.
- `TASK_ACCEPTED`는 최소 `job_id`, `control_seq`, `dispatch_id`를 포함해야 합니다.
- `TASK_DONE`은 최소 `job_id`, `control_seq`, `dispatch_id`를 포함해야 합니다.
- `BROKEN`은 최소 `reason`을 포함해야 합니다.

## 4. 비기능 요구사항

## NFR-001. 24시간 안정성 (채택 baseline)
시스템은 24시간 무인 soak test에서 운영 중단 없이 동작해야 합니다.
이 요구사항은 **채택(adoption) / baseline gate 1회 통과 기준**이며, 채택 이후의 day-to-day 검증 authority는 `05_운영_RUNBOOK.md §3.5`(launcher live stability gate + incident replay + 실제 작업 세션)입니다.

**수용 기준 (채택 baseline 1회 수행 기준)**
- manual intervention 0회
- duplicate dispatch 0건
- receipt 누락 0건
- false READY / false CLOSED 0건

**적용 범위 메모**
- 이 수치는 채택 전 baseline 또는 runtime 계약이 크게 바뀐 경우의 adoption-final gate에서 요구합니다.
- 일상 운영에서 24h soak를 반복 재실행하는 것은 기본 검증 메뉴가 아닙니다.

## NFR-002. 복구 가능성
단일 lane crash, watcher crash, stale lock, corrupt state는 전체 runtime 장애로 바로 이어지지 않아야 합니다.

**수용 기준**
- lane 단위 고장 격리
- 안전정지 또는 자동복구 제공

## NFR-003. 일관성
동일 시점의 상태는 어디서 보더라도 동일해야 합니다.

**수용 기준**
- supervisor 외의 계산 경로가 current truth를 만들지 않습니다.
- control block과 lane/round 상태가 서로 다른 근거로 갈라지지 않습니다.

## NFR-004. 성능
status 조회는 가벼워야 하며, 파일시스템 전체 스캔을 반복하지 않아야 합니다.

**권고 기준**
- 상태 조회 p95 200ms 이하
- health polling 주기 1~2초에서도 I/O 부하가 급증하지 않음

## NFR-005. 운영성
장애 분석에 필요한 정보가 충분해야 합니다.

**수용 기준**
- run_id 단위 로그 추적 가능
- recent events 확인 가능
- degraded reason 확인 가능
- active control 확인 가능

## NFR-006. 변경 안전성
shipped browser contract와 독립적으로 배포/회귀 검증할 수 있어야 합니다.

**수용 기준**
- internal pipeline tooling 회귀가 `/app` 기본 계약 회귀와 분리됩니다.

## NFR-007. 테스트 가능성
모든 핵심 상태 전이는 자동 테스트 또는 재현 스크립트로 검증 가능해야 합니다.

**수용 기준**
- unit / integration / soak / fault injection 항목이 문서화됩니다.
- shadow diff와 takeover 검증 항목이 포함됩니다.

- `BROKEN`

## 5.2 Lane 상태
- `OFF`
- `BOOTING`
- `READY`
- `WORKING`
- `DEGRADED`
- `BROKEN`
- `STOPPED`

## 5.3 Round 상태
- `DISCOVERED`
- `STABILIZING`
- `DISPATCHED`
- `VERIFYING`
- `RECEIPT_PENDING`
- `CLOSED`
- `FAILED`
- `WAITING_FEEDBACK`

## 5.4 Control surface 요구사항
- active control은 `CONTROL_SEQ` 우선, 필요 시 `mtime` 보조 기준으로 계산합니다.
- status는 현재 활성 control file과 sequence를 명시적으로 노출해야 합니다.
- stale control 후보가 현재 active처럼 보이면 안 됩니다.

## 6. 결론

목표는 단순한 backend 교체가 아니라 **상태·제어·산출물의 경계 고정**입니다.  
특히 compatibility 범위, control block, migration ownership, wrapper event 계약이 고정되지 않으면 구현 단계에서 다시 모호해질 가능성이 큽니다.
