# projectH Pipeline Runtime 기술설계 명세서
버전: v1.1  
작성일: 2026-04-10  
문서 성격: 목표 아키텍처 / 인터페이스 명세

## 1. 설계 목표

이번 설계의 목적은 다음 여섯 가지입니다.

1. launcher를 상태 추론기에서 명령 클라이언트로 축소
2. supervisor를 유일한 lifecycle / state authority로 승격
3. status v1에 control block을 포함해 기존 운영 화면을 대체
4. tmux를 adapter 뒤로 숨겨 backend detail로 제한
5. run-scoped state와 receipt를 도입해 restart/recovery 일관성 확보
6. wrapper event를 공식 readiness 근거로 고정

## 2. 아키텍처 개요

```text
[Launcher / GUI / Controller]
          |
          v
     [Supervisor]
       /   |    \
      /    |     \
 [Status] [Events] [Commands]
    |                 |
    v                 v
 [Run Store]     [Backend Adapter]
                      |
               +------+------+
               |             |
           [TmuxAdapter]  [PtyAdapter(optional)]
               |
               v
          [Lane Wrapper]
               |
               v
          [Vendor CLI]
               |
               v
      [Watcher / Orchestrator]
```

## 3. 컴포넌트 책임

## 3.1 Launcher
### 책임
- start/stop/restart/attach 요청
- status 조회 및 렌더링
- 최근 이벤트의 읽기 전용 표시
- shared runtime helper/CLI를 통한 로컬 thin-client 경계 유지

### 책임 아님
- pane 출력 직접 해석
- watcher.log 직접 해석
- `/work`, `/verify` 파일 스캔으로 현재 상태 판정
- tmux 세션 유무만으로 readiness 결정
- raw control slot 파일 해석으로 active control 재계산

## 3.2 Supervisor
### 책임
- current run lifecycle 관리
- backend 시작/종료
- lane 상태 집계
- active control 계산 및 status 반영
- receipt 기록
- degraded/recovery 전이
- 상태 파일/API 제공
- 같은 `project_root + session` 범위의 duplicate supervisor 정리 및 단일 writer 유지

### 금지
- 여러 외부 consumer가 상태를 수정하도록 허용
- artifact 존재만으로 round 종료 처리

## 3.3 Backend Adapter
### 책임
- 실제 프로세스/세션 substrate 관리
- lane spawn/terminate/restart
- health probe
- attach entrypoint 제공
- lane wrapper 실행 및 wrapper event 수집

### 비책임
- round business logic
- verify policy
- next slice 선택

## 3.4 Watcher/Orchestrator
### 책임
- artifact 관찰
- dispatch 정책
- dedupe
- lease
- verify 결과 처리
- manifest 검증
- receipt 후보 전달

### 비책임
- UI 상태 렌더링
- launch 상태 추론 화면
- production status writer

## 4. 인터페이스 명세

## 4.1 Supervisor Command API
형식은 로컬 HTTP, Unix socket, 파일 queue 중 아무 방식이든 가능하나, 의미는 아래를 따라야 합니다.

### `POST /runtime/commands/start`
요청 예시:
```json
{
  "requested_by": "launcher",
  "profile": "default",
  "backend": "tmux"
}
```

응답 예시:
```json
{
  "accepted": true,
  "run_id": "20260410T103000Z-001",
  "runtime_state": "STARTING"
}
```

### `POST /runtime/commands/stop`
```json
{
  "requested_by": "launcher",
  "reason": "operator_request"
}
```

- stop command는 supervisor에 먼저 `SIGTERM`을 보내고, final status가 `runtime_state=STOPPED`로 flush될 때까지 bounded wait를 합니다.
- graceful flush가 timeout 안에 끝나지 않을 때만 강제 종료 fallback을 사용합니다. 이 fallback은 success처럼 포장하지 않고, orphan watcher/tmux session이 남았으면 cleanup 뒤 `status.json`을 inactive truth(`control=none`, `active_round=null`, watcher dead, lane `OFF`)로 강제 보정합니다.

### `POST /runtime/commands/restart`
```json
{
  "requested_by": "launcher",
  "reason": "operator_request"
}
```

### `POST /runtime/commands/attach`
```json
{
  "requested_by": "launcher",
  "lane": "claude"
}
```

### 설계 규칙
- status v1은 현재 운영 화면에 필요한 control 정보를 포함해야 합니다.
- `active_control_file`과 `active_control_seq`는 status의 일부이며 optional field가 아닙니다.
- `autonomy` block은 `runtime_state`와 별개인 unattended runtime 판단 surface입니다. `mode = normal|recovery|triage|hibernate|pending_operator|needs_operator`를 사용하고, `needs_operator` current truth를 바로 publish하지 못한 동안의 gate 상태를 여기에 남깁니다.
- launcher/GUI는 control slot 원문을 직접 파싱하지 않고 이 block을 사용합니다.
- authority path는 `.pipeline/runs/<run_id>/status.json`과 `.pipeline/runs/<run_id>/events.jsonl`입니다.
- `.pipeline/state/turn_state.json`는 migration 동안만 export-only mirror입니다.
- control-plane arbitration에서 next-slice ambiguity가 유일한 blocker면 Codex는 `.pipeline/operator_request.md`보다 `.pipeline/gemini_request.md`를 먼저 열어야 합니다.
- `.pipeline/operator_request.md`는 real operator-only decision, approval/truth-sync blocker, immediate safety stop, 또는 Gemini advice 이후에도 exact slice가 남지 않을 때만 surface 합니다. canonical stop slot은 `STATUS`, `CONTROL_SEQ`, `REASON_CODE`, `OPERATOR_POLICY`, `DECISION_CLASS`, `DECISION_REQUIRED`, `BASED_ON_WORK`, `BASED_ON_VERIFY`를 top header로 유지합니다.
- supervisor/watcher는 operator stop publish/gate를 `OPERATOR_POLICY` 우선, `REASON_CODE` 다음, prose 마지막 참고 순서로 판정합니다.
- runtime-owned/scripted stop slot write는 정책을 다시 계산하지 않고 `pipeline_runtime.control_writers.write_operator_request(...)`로 필수 header를 검증한 뒤 atomic write만 수행하는 편이 맞습니다.
- `safety_stop`, `approval_required`, `truth_sync_required`는 즉시 current truth operator stop으로 올리고, `context_exhaustion`, `session_rollover`, `continue_vs_switch`, `slice_ambiguity`, `auth_login_required`, `provider_outage`, `receipt_repair`, `verify_manifest_mismatch`, `duplicate_handoff`, `lane_recovery_exhausted`, `session_missing` 같은 non-safety blocker는 최대 24시간 `autonomy` gate 뒤에 둡니다.
- structured metadata가 없거나 알 수 없으면 fail-safe로 즉시 publish합니다. 24시간 gate는 structured `REASON_CODE`/`OPERATOR_POLICY`가 분명한 stop slot에만 적용합니다.
- stale operator stop이 suppression으로 `control=none`이 되면 watcher는 `CODEX_FOLLOWUP` turn으로 Codex control-recovery를 1회 재개할 수 있습니다.
- 이 control-recovery는 stale stop 자체를 삭제하지 않고, Codex가 다음 canonical control(`claude_handoff` / `gemini_request` / `operator_request`) 중 정확히 하나를 다시 쓰게 하는 bounded follow-up입니다.
- 브라우저 controller UI는 controller HTTP API만 소비합니다.
- controller browser의 active HTTP contract는 `/api/runtime/status`, `/api/runtime/start|stop|restart`, `/api/runtime/capture-tail`, `/api/runtime/send-input`로 제한합니다.
- controller browser는 `runtime_state`가 `STOPPED`, `STOPPING`, `BROKEN`이면 live `control`, `active_round`, lane action을 active처럼 surface하면 안 됩니다.
- controller browser는 `runtime_state=DEGRADED`이면서 `degraded_reason`이 `supervisor_missing_recent_ambiguous` 또는 `supervisor_missing_snapshot_undated`인 incomplete snapshot을 uncertain runtime으로 표시합니다. 이 경우 badge는 amber `DEGRADED`, `Control`/`Round`는 `uncertain`, `Watcher`는 `Unknown`으로 내려가야 하며 raw lane state는 read-model 시각화용으로만 유지합니다.
- 브라우저 controller UI는 동일 runtime API를 읽는 optional `Office View` 시각화 토글을 둘 수 있지만, 이 뷰 역시 status/tail의 read-model일 뿐 별도 상태 authority가 아닙니다.
- `Office View`는 외부 레퍼런스 복제가 아니라 projectH 전용 `3-lane war-room` read-model로 유지합니다. Claude / Codex / Gemini는 동등한 3개 station으로 보이고, watcher는 room 안의 ops core object로만 표현됩니다.
- `Office View`는 optional operator-supplied animation assets를 `controller/assets/` 아래에서 읽을 수 있지만, 자산 부재 시에도 기존 fallback rendering으로 동작해야 합니다.
- 상태별 GIF(`BOOTING.gif`, `WORKING.gif`, `BROKEN.gif`, `READY.gif`, `DEAD.gif`)가 있으면 browser는 이를 우선 사용하고, 없거나 아직 로드되지 않았을 때만 `scripts/build_office_sprites.py`가 생성한 `controller/assets/generated/office-sprite-manifest.json` + normalized PNG frame set로 fallback 합니다.
- Office View background는 `/controller-assets/background.png`를 우선 사용하되, cached/onload race 또는 root asset 부재 시 `/controller-assets/generated/bg-office.png`로 fallback 할 수 있어야 하며, browser는 `Scene` read-model로 `loading` / `root` / `fallback` / `asset_error`를 operator에게 보여줄 수 있습니다.
- raw sprite sheet crop/trim 경로는 generated PNG 자산까지 없을 때의 최후 fallback으로만 남깁니다.
- opaque 배경의 설명용 sprite sheet를 사용할 때는 controller browser가 client-side frame crop / edge-whitespace trim / frame animation만 수행할 수 있습니다. 이 처리는 read-model 시각화용에 한정되며 runtime state 계약을 바꾸지 않습니다.
- Office View sprite animation은 browser에서 frame viewport를 정규화하고 ping-pong sequence / light crossfade를 적용해, 프레임별 크기 점프와 hard reset 느낌을 줄일 수 있습니다.
- Office View의 handoff packet, weather tint/rain, office pet, ambient audio 같은 연출은 모두 기존 runtime payload(`control`, `latest_work`, `latest_verify`, `last_receipt`, lane state)를 읽는 decorative read-model에 한정됩니다. browser가 pane text parsing이나 별도 상태 추론으로 이를 생성하면 안 됩니다.
- `latest_work`, `latest_verify`는 각각 `work/<month>/<day>/YYYY-MM-DD-*.md`, `verify/<month>/<day>/YYYY-MM-DD-*.md` 형식의 canonical round note만 가리켜야 하며, `work/README.md`, `verify/README.md` 같은 metadata 문서를 최신 artifact로 노출하면 안 됩니다.
- working / booting lane은 desk anchor를 기준으로 보이되, `ready` / `idle` 같은 non-working lane은 browser-local predefined room floor anchor들 사이를 roam 할 수 있습니다. 이 wandering은 purely decorative read-model이며 runtime lane truth나 authority를 바꾸지 않습니다.
- ambient audio는 browser-local UX여야 하며 explicit user gesture 이후에만 시작할 수 있습니다. 첫 `poll()` 성공만으로 자동 재생을 시작하면 안 됩니다.
- Office View는 browser-local reduced-motion 토글(`✨` 버튼)을 제공합니다. 활성화하면 weather rain, pet roaming, event particle, delivery packet animation을 멈추고, agent 렌더링 · runtime badge · event log · log modal · needs-operator overlay는 그대로 유지합니다. 이 설정은 backend route를 추가하지 않으며 순수 browser-local입니다.
- reduced-motion 토글과 ambient mute 토글의 상태는 `localStorage`(`office_low_motion`, `office_muted`)에 저장되어 브라우저 새로고침 후에도 유지됩니다. 복원 시 muted 상태는 audio를 자동 재생하지 않고 아이콘/title만 반영합니다.
- 모든 browser-local preference 접근은 `PrefStore` 공유 helper를 통해 이루어집니다. `PrefStore`는 최초 접근 시 한 번만 `localStorage` 가용성을 검사하고, 차단된 환경(private browsing 등)에서는 event log에 `환경 설정 저장 불가` 경고를 표시하여 operator가 persistence 비활성 상태를 인지할 수 있게 합니다. 이 신호는 non-authoritative이며 runtime authority를 변경하지 않습니다.
- `localStorage`가 차단된 환경에서는 event log 경고 외에 toolbar 영역에 `⚠ 설정 비저장` 경고 chip(`#storage-warn`)이 표시됩니다. tooltip은 `localStorage 사용 불가 — 새로고침 시 toolbar 설정이 초기화됩니다`를 안내합니다. 이 chip은 `PrefStore.available`이 false일 때만 `display`가 활성화되며, runtime authority와 무관한 non-authoritative 시각 안내입니다.
- 브라우저 controller UI는 typed operator command palette를 둘 수 있지만, 이는 기존 controller HTTP API에 매핑되는 bounded command만 허용하며 arbitrary shell exec를 열지 않습니다.
- browser log modal은 tail read와 bounded one-line lane input만 제공합니다. 이 입력은 permission/plan prompt 같은 interactive lane prompt에 `1`, `2`, `3` 등 짧은 응답을 보내는 용도이며, arbitrary shell exec가 아닙니다. backend route가 없는 lane pause/resume/restart나 attach affordance는 계속 숨기거나 disabled 상태로 유지해야 합니다.
- Codex lane wrapper는 startup 중 self-update dialog를 runtime prompt로 오인하면 안 됩니다. `Update available` / `Update now` / `Skip until next version` 류의 blocking prompt가 보이면 unattended runtime에서는 `Skip until next version`을 먼저 선택하고 `READY(prompt_visible)`를 emit하지 않아야 합니다.
- `pipeline_gui`와 `pipeline-launcher`는 로컬 thin client로서 shared runtime status/event helper와 CLI를 사용합니다.
- thin client current truth 우선순위는 `runtime status` canonical block -> `turn_state` current turn authority -> current verify `active_round` 순서이며, pane/log/file scan은 drift detection 보조 경로로만 유지합니다.
- 어떤 consumer도 pane/log/file scan을 current truth 경로로 재도입하지 않습니다.
- `start` / `restart` 경로는 같은 `project_root + session`의 live supervisor를 먼저 reconcile해야 하며, 여러 daemon이 동시에 `current_run.json`과 `status.json`을 덮어쓰는 상태를 허용하지 않습니다.
- live daemon이 1개만 남아 있는데 pidfile이 stale하면 CLI는 pidfile을 그 daemon pid로 복구해야 합니다.
- `status.control`이 `none`이어도 `turn_state=CODEX_FOLLOWUP`이면 Codex lane은 `WORKING` / `note=followup`으로 surface될 수 있습니다. 이는 stale operator stop recovery 같은 no-active-control follow-up 경계를 위한 예외입니다.

## 5. 디렉터리 구조

```text
.pipeline/
  claude_handoff.md
  gemini_request.md
  gemini_advice.md
  operator_request.md
  current_run.json
  state/
    turn_state.json
    autonomy_state.json
  runs/
    <run_id>/
      status.json
      events.jsonl
      logs/
        supervisor.log
        backend.log
        watcher.log
      state/
        jobs/
      locks/
      manifests/
      receipts/
        <job_id>-r<round>.json
      wrapper-events/
        claude.jsonl
        codex.jsonl
        gemini.jsonl
      backend/
        tmux.json
      compat/
        legacy-status.json
        latest-work.txt
        latest-verify.txt
```

### 설계 의도
- 현재 run과 이전 run을 명확히 분리합니다.
- legacy surface가 필요하면 `compat/` 아래 export-only로 둡니다.
- `compat/`는 read model만 담고, rolling control slot은 복제하지 않습니다.
- rolling control slot은 `.pipeline/` 루트에 남기되, status는 그 active 결과만 surface 합니다.
- 전역 `.pipeline/state/*` 직접 삭제/재생성 모델을 제거합니다.
- 공유 `.pipeline/state/`는 `turn_state.json`, `autonomy_state.json`, 그리고 blocker resolution에 아직 필요한 `VERIFY_DONE` 기록만 오래 남길 수 있습니다. 이전 run의 non-terminal job state는 startup cleanup에서 각 run의 `state-archive/` 또는 legacy archive로 옮겨 현재 run status와 섞이지 않게 합니다.
- supervisor-managed runtime에서는 watcher job state도 supervisor가 발급한 동일 `run_id`를 사용합니다. watcher가 독자 `run_id`를 쓰면 shared `.pipeline/state`의 current job state가 current run status에서 누락될 수 있으므로 허용하지 않습니다.
- lane `READY/WORKING` surface는 wrapper event를 primary로 유지하되, attachable pane의 tail 하단 prompt/activity marker는 stale READY/WORKING drift를 좁게 보정하는 secondary evidence로만 씁니다. current `accepted_task`가 살아 있는 lane은 matching `TASK_DONE`/BROKEN/inactive 전이 전까지 tail prompt만으로 `READY`로 내려가면 안 됩니다.
- pane busy/ready/prompt-visible/tail-state 판정은 `pipeline_runtime/lane_surface.py` shared helper/profile을 single source로 두는 편이 맞습니다. watcher, supervisor, cli wrapper가 각자 marker 집합을 오래 따로 들고 있으면 thin-client drift가 다시 커집니다.
- thin client snapshot은 verify lane에서 `active_round.state`, `dispatch_id`, `completion_stage`, receipt pending 여부를 함께 보여 current runtime truth를 설명해야 합니다. pane text 자체를 current truth로 재판정하는 helper는 추가하지 않습니다.
- `RECEIPT_PENDING`는 "receipt close authority가 아직 안 닫혔다"는 뜻이지, current turn이 이미 follow-up/advisory/idle로 넘어간 뒤에도 verify owner를 계속 active lane으로 고정하라는 뜻은 아닙니다. 이 상태는 기본적으로 `active_round`와 lane note로 surface하고, active lane 우선순위는 current turn/control을 먼저 따르는 편이 맞습니다.
- metadata-only `/work` closeout도 latest canonical round note라면 watcher의 Claude round-completion 감지에는 포함합니다. 따라서 watcher는 `CLAUDE_ACTIVE` progress snapshot과 verify artifact 후보를 broad latest note 기준으로 한 번 더 보되, dispatch prompt content를 좁히는 용도로만 metadata-only filtering을 유지합니다.
- latest `/work` verify 필요 판정도 generic latest same-day `/verify` mtime만으로 닫지 않습니다. watcher는 current run `VERIFY_DONE` state 또는 verify note 본문 안의 `work/...md` 참조로 latest `/work`와 실제로 연결된 matching `/verify`를 먼저 찾고, `verify/<month>/<day>/` day-folder 바로 아래 note도 canonical same-day verify로 인정합니다. unrelated newer `/verify`가 있더라도 latest `/work`가 아직 매칭 verify 없이 남아 있으면 Codex verify를 우선 유지합니다. `REASON_CODE: newer_unverified_work_present` + `OPERATOR_POLICY: stop_until_truth_sync`는 immediate operator stop이 아니라 same-family truth-sync recovery로 분류해 Codex verify 우선 경계와 맞춥니다.

## 6. 상태 모델

## 6.1 Runtime state machine
```text
STOPPED
  -> STARTING
  -> RUNNING
  -> DEGRADED
  -> STOPPING
  -> STOPPED

RUNNING
  -> BROKEN (복구 실패 시)
DEGRADED
  -> RUNNING (복구 성공 시)
```
- 정상 stop 성공 기준은 supervisor 프로세스 종료만이 아니라 final status flush 완료입니다. supervisor는 종료 `finally`에서 `runtime_state=STOPPED`, `control.active_control_status=none`, `active_round=null`, watcher inactive, lane `OFF` 상태를 직접 기록한 뒤에만 stop 성공으로 간주됩니다.
- supervisor가 이미 내려간 오래된 `STOPPING` status를 reader가 다시 만나면 browser/controller surface에서는 이를 `STOPPED`로 정규화하고, live `control` / `active_round`는 비웁니다. 이 reader 정규화는 primary truth가 아니라 stale-read safety net입니다.
- `RUNNING` 또는 `DEGRADED` snapshot 이 아직 recent 하더라도, supervisor pid 가 없고 status 안에 기록된 watcher/lane pid 들도 이미 quiescent 면 reader 는 stale timeout 을 기다리지 않고 즉시 `BROKEN(supervisor_missing)` 으로 강등할 수 있습니다.
- pid 가 비어 있더라도 field 자체가 이미 `control=none`, `active_round=null`, watcher inactive, lane `OFF/BROKEN/DEAD` 로 정리된 recent snapshot 이면 reader 는 같은 fast-path 로 stale `RUNNING` 착시를 줄일 수 있습니다.
- recent snapshot 이고 supervisor 가 없지만, `control != none` / `active_round != null` / active lane state 같은 activity claim 은 남아 있는 반면 watcher 와 active lane pid 로는 live identity 를 증명하지 못하면 reader 는 이를 `DEGRADED(supervisor_missing_recent_ambiguous)`로 정규화합니다.
- 같은 ambiguous payload 에 `updated_at`까지 비어 있어 recent/stale 판단 근거가 없으면 reader 는 이를 `DEGRADED(supervisor_missing_snapshot_undated)` uncertain runtime 으로 정규화합니다.
- 이 ambiguous normalization 은 raw `control`, `active_round`, `lanes`, `watcher` field 를 지우지 않고 보존합니다. stale timeout 이 지나면 그때 기존 `BROKEN(supervisor_missing)` inactive cleanup 으로 내려갑니다.

## 6.2 Lane state rules
- `BOOTING`: 프로세스/세션이 생성되었으나 wrapper `READY` event 미도착
- `READY`: wrapper가 입력 가능 상태를 명시적으로 보고
- `WORKING`: wrapper가 `TASK_ACCEPTED` 또는 active work를 보고
- `DISPATCH_SEEN`은 wrapper가 active task hint를 실제로 읽은 pre-accept 관측점이며, lane state를 바로 `WORKING`으로 올리지 않고 note/diagnostic용으로만 유지합니다.
- `TASK_DONE`은 `TASK_ACCEPTED` 이후 current dispatch가 끝났다는 선행 완료 신호이며, read model은 `done_task`와 `done_at`으로 이를 직접 유지합니다.
- `DEGRADED`: 프로세스는 살아 있으나 health anomaly 존재
- `BROKEN`: 프로세스 또는 backend recover 실패
- active lane pane tail에 `401`, `Invalid authentication credentials`, `Please run /login` 같은 auth/login failure가 보이면 supervisor는 해당 lane을 `BROKEN`, note를 `auth_login_required`로 surface하고 blind restart를 시도하지 않습니다.
- duplicate/no-op implement handoff가 감지되면 Claude lane은 `WORKING`에 고정되지 않고 `READY`로 내려가며 note는 `waiting_next_control`을 사용합니다.

## 6.3 Round state rules
- artifact 발견만으로 `DISPATCHED`가 되지 않습니다. dispatch event가 있어야 합니다.
- verify round close는 `TASK_ACCEPTED -> TASK_DONE -> current-round /verify change + next control change + receipt` 순서를 만족해야 합니다.
- verify 산출물이 있어도 matching `TASK_DONE`나 receipt가 없으면 `CLOSED`가 아닙니다.
- verify refutation이 오면 기존 round는 `FAILED` 또는 `WAITING_FEEDBACK`으로 남고, 새 round가 열립니다.
- verify round는 active control slot 없이도 현재 dispatch truth를 유지해야 하므로 `active_round.dispatch_control_seq`를 별도 필드로 유지합니다. Codex task hint는 `active_round.job_id + dispatch_id`가 존재하면 active가 될 수 있고, `status.control=none`만으로 inactive 처리하면 안 됩니다.
- `CODEX_FOLLOWUP` 또는 `GEMINI_ADVISORY`로 턴이 넘어간 뒤에는 stale verify `active_round`를 current status surface에 남기지 않습니다. 이때 Codex/Gemini hint는 current control seq 기반으로만 active가 되고, 이전 verify `job_id`/`dispatch_id`는 task hint와 status 둘 다에서 비워야 합니다.
- wrapper `TASK_DONE`는 task hint clear 이후가 아니라, accepted된 current dispatch가 prompt-visible + no-busy로 안정된 시점에 먼저 emit될 수 있어야 합니다.

## 6.4 Control rules
- active control은 `CONTROL_SEQ` 우선, 필요 시 `mtime`을 보조 기준으로 계산합니다.
- stale control 후보가 존재해도 status에는 active 1개만 노출합니다.
- control 계산 로직은 supervisor 또는 supervisor가 위임한 reader 한 곳에만 존재합니다.
- 이미 완료된 same-handoff SHA가 다시 active implement control로 남아도 runtime은 이를 다시 active work로 승격하지 않습니다.
- 이 경우 supervisor는 `control_duplicate_ignored`를 기록하고, Canonical `.pipeline/claude_handoff.md`는 그대로 두되 Claude lane surface만 `READY + waiting_next_control`로 정규화합니다.
- `.pipeline/operator_request.md`가 active여도, 본문에 적힌 blocker `work/...md` 경로들이 이미 run state에서 모두 `VERIFY_DONE`으로 닫혔으면 supervisor는 이를 stale operator stop으로 취급합니다.
- 이 경우 canonical `.pipeline/operator_request.md` 파일은 그대로 두되, runtime `status.control`은 `none`으로 내리고 `events.jsonl`에는 `control_operator_stale_ignored`를 기록합니다.
- `.pipeline/operator_request.md`가 fresh active여도, 본문 reason이 approval/truth-sync/safety가 아니면 supervisor는 이를 먼저 gated operator candidate로 분류합니다.
- gate 중에는 canonical 파일은 그대로 두되, runtime `status.control`은 `none`, `status.autonomy.mode`는 `recovery|triage|hibernate|pending_operator`, `events.jsonl`에는 `control_operator_gated`를 기록합니다.
- idle stable 상태에서 gated candidate가 더 할 일이 없으면 runtime은 operator wait 대신 `autonomy.hibernate`로 남고, thin client는 이를 active but unattended 상태로만 보여야 합니다.
- supervisor가 사라져 runtime status가 stale `BROKEN`으로 강등된 경우에도 browser/controller가 live control이나 active round를 계속 active처럼 surface하면 안 되므로, reader는 `status.control=none`, `active_round=null`로 정규화합니다.
- supervisor가 이미 사라진 raw `BROKEN` snapshot 도 current truth 로는 live round/control 을 가질 수 없으므로, reader 는 `status.control=none`, `active_round=null`, watcher inactive, lane pid 제거를 함께 정규화합니다.
- ambiguous recent incomplete snapshot 은 current truth 가 uncertain 한 경계이므로 browser/controller 는 raw lane animation 보다 sidebar/runtime truth 를 우선해 보여야 합니다. 즉 lane state 는 그리더라도 runtime summary 는 clean `RUNNING`처럼 보이면 안 됩니다.

### 스키마 예시

### 규칙
- receipt는 단 한 번만 최종 close를 선언합니다.

- receipt와 manifest hash가 불일치하면 `DEGRADED`로 전이합니다.
- receipt는 runtime close authority이며, `/work`와 `/verify` 자체를 대체하지 않습니다.
- `TASK_DONE`는 receipt close permission의 선행 신호이고, watcher/wrapper는 completion fact를 먼저 surface한 뒤 supervisor가 receipt로 최종 close를 선언합니다.
- `/work`와 `/verify`는 persistent truth / audit artifact로 남습니다.
- round-close receipt는 supervisor만 발급합니다.
- watcher, wrapper, Codex/Gemini lane은 receipt 대신 completion fact만 보고합니다.

## 10. Wrapper / Sentinel Event 설계

## 10.1 배치 위치
- `LaneWrapper`는 adapter가 launch하는 각 lane command의 immediate parent process입니다.
- wrapper는 vendor CLI child를 생성하고 stdin/stdout/exit를 감시합니다.
- supervisor는 wrapper가 남긴 공식 이벤트만 소비합니다.

## 10.2 수집 경로
초기 구현에서는 아래 경로를 고정합니다.

```text
.pipeline/runs/<run_id>/wrapper-events/<lane>.jsonl
```

- lane별로 파일 1개만 씁니다.
- writer는 해당 lane wrapper 1개뿐입니다.
- supervisor는 이 파일을 tail 하거나 fs-notify로 구독합니다.
- 나중에 socket transport로 바꾸더라도 event schema는 유지합니다.

## 10.3 필수 이벤트
```json
{
  "ts": "2026-04-10T10:35:20.100Z",
  "run_id": "20260410T103000Z-001",
  "lane": "claude",
  "event": "READY",
  "derived_from": "wrapper_handshake",
  "detail": {}
}
```

필수 event:
- `READY`
- `HEARTBEAT`
- `DISPATCH_SEEN`
- `TASK_ACCEPTED`
- `TASK_DONE`
- `BROKEN`

필드 규칙:
- `derived_from`은 `process_lifecycle`, `wrapper_handshake`, `vendor_output` 중 하나입니다.
- launcher/GUI는 raw pane text를 직접 보지 않고 이 event 결과만 소비합니다.
- `DISPATCH_SEEN.detail`에는 최소 `job_id`, `control_seq`가 포함됩니다.
- `DISPATCH_SEEN.detail.dispatch_id`, `TASK_ACCEPTED.detail.dispatch_id`, `TASK_DONE.detail.dispatch_id`는 current dispatch direct matching에 사용됩니다.
- `TASK_ACCEPTED.detail`에는 최소 `job_id`, `control_seq`, `dispatch_id`가 포함됩니다.
- `TASK_DONE.detail`에는 최소 `job_id`, `control_seq`, `dispatch_id`가 포함됩니다.
- verify task hint는 active control slot이 없어도 `active=true`, `job_id`, `dispatch_id`, round-scoped `control_seq`를 포함해야 합니다. supervisor는 `active_round.dispatch_control_seq`를 우선 채우고, wrapper는 이 active hint를 control slot 부재만으로 무시하지 않습니다.
- follow-up/advisory task hint는 current control-driven 경계이므로 stale verify `job_id`/`dispatch_id`를 carry-over하지 않습니다. `CODEX_FOLLOWUP` 상태에서 이전 verify metadata를 다시 실으면 wrapper bridge와 runtime surface가 current truth를 잘못 섞게 됩니다.
- follow-up/advisory/operator/blocked-triage prompt dispatch는 lane이 실제 prompt-ready로 안정됐을 때만 paste하고, busy lane이면 runtime event `lane_input_deferred`를 남긴 채 pending defer로 유지합니다. thin client는 이 진단 이벤트를 그대로 보여주되 status truth를 덮어쓰지 않습니다.
- queue/defer/flush/drop 구현은 `watcher_dispatch.py` 공용 경계에 두고, `watcher_core.py`는 prompt 조립과 dispatch intent 생성까지만 담당하는 편이 맞습니다. 그래야 notify family마다 send semantics가 다시 갈라지지 않습니다.
- pending defer는 current active control과 같은 family/seq가 살아 있을 때만 유지합니다. active control이 더 높은 seq handoff나 다른 family로 넘어가면 예전 pending prompt는 flush하지 않고 drop해야 하며, 그렇지 않으면 Claude 차례 직전에 stale Codex follow-up이 잘못 paste될 수 있습니다.
- watcher의 busy/prompt-ready 판정은 pane 전체 scrollback이 아니라 최근 visible tail에 한정해 보는 편이 맞습니다. 예전 `Working (...)` 줄이 위에 남아 있다는 이유만으로 현재 prompt-visible pane을 계속 busy lane으로 분류하면 wrong defer가 반복됩니다.
- background terminal wait 같은 explicit busy tail이 current pane에 남아 있는 동안에는 wrapper/watcher가 post-accept active wait를 유지해야 하며, `TASK_DONE` missing deadline을 그대로 false stall로 승격하면 안 됩니다.
- `TASK_DONE.detail.reason`은 선택 필드이며 `task_hint_cleared`, `duplicate_handoff`, `blocked_triage`를 사용할 수 있습니다.
- `BROKEN.detail`에는 최소 `reason`이 포함됩니다.
- hint가 active인데 `control_seq` metadata가 누락되었거나 비정상이면 wrapper는 silent no-event 대신 `BRIDGE_DIAGNOSTIC(code=active_task_hint_metadata_invalid)`를 남깁니다.

## 10.4 임시 parser 허용 범위
- vendor CLI가 explicit handshake를 주지 않으면, **wrapper 내부에서만** vendor 출력 기반 parser를 둘 수 있습니다.
- 이 parser는 `derived_from: "vendor_output"`으로 표시된 공식 wrapper event만 만들어야 합니다.
- TmuxAdapter, launcher, GUI, watcher가 각자 별도의 parser를 두는 것은 금지합니다.
- parser는 `READY`, `DISPATCH_SEEN`, `TASK_ACCEPTED`, `TASK_DONE` 같은 lane read model 보조에만 사용합니다.
- round close, receipt 확정, control 결정은 parser 결과만으로 하면 안 됩니다. 다만 verify close permission은 matching `TASK_DONE`를 반드시 선행 조건으로 사용합니다.
- stale degraded/autonomy surface도 current round 기준으로만 유지합니다. runtime이 `STOPPED`이거나 `active_round.artifact_path`가 바뀌면 이전 round의 `dispatch_stall`, 오래된 `/work` 기반 autonomy warning은 current truth에서 제거해야 합니다.

## 10.5 Recovery policy
- Claude implement lane은 `TASK_ACCEPTED` 이전 종료 시 1회 자동 재기동과 동일 task 재전송을 허용합니다.
- Claude implement lane은 `TASK_ACCEPTED` 이후 종료 시 blind replay를 금지하고 `INTERRUPTED` 의미의 follow-up 판단으로 넘깁니다.
- Codex verify lane은 `/verify` 업데이트와 round-close receipt가 아직 없으면 2회까지 자동 재기동과 동일 verify 재시도를 허용합니다.
- Gemini advisory lane은 advisory completion fact가 아직 없으면 2회까지 자동 재기동과 동일 advisory 재시도를 허용합니다.
- supervisor/watcher 재시작 후에는 persisted state를 읽어 복구하되 dedupe key로 중복 dispatch를 막습니다.

## 11. 전환 절차

### Step 1. status schema v1 + temporary ownership 규칙 고정
- production `status.json`의 schema와 control block을 먼저 고정합니다.
- 이 단계의 production status writer는 watcher-exporter 1개만 허용합니다.
- launcher/GUI는 새 status만 읽도록 수정합니다.
- raw tmux/log/file scan UI는 이 단계에서 제거하거나 숨깁니다.

### Step 2. supervisor shadow mode 도입
- supervisor는 동일 입력을 바탕으로 shadow status를 별도 경로에 기록합니다.
- shadow output은 diff 용도이며 production writer가 아닙니다.
- watcher-exporter와 supervisor shadow 결과를 비교해 불일치를 수집합니다.

### Step 3. supervisor takeover
- supervisor를 production status writer로 승격합니다.
- watcher-exporter는 같은 변경 세트에서 즉시 비활성화합니다.
- takeover 시점부터 `status.json` current truth writer는 supervisor 하나입니다.

### Step 4. tmux 직접 호출 제거
- launcher에서 tmux 명령 제거
- TmuxAdapter 안으로 흡수
- status/control 계산 로직도 supervisor 쪽으로 집중

### Step 5. run-scoped state
- `runs/<run_id>` 구조 도입
- legacy status path는 export-only 유지 가능
- rolling control slot은 계속 root `.pipeline/`에 유지

### Step 6. receipt 도입
- round close 표시를 receipt로 변경
- `/work`, `/verify`는 persistent artifact로 유지

### Step 7. wrapper contract 활성화
- lane별 wrapper 배치
- wrapper-events 수집 시작
- STARTING/READY/WORKING 판정을 wrapper event 기준으로 전환

### Step 8. recovery / fault-tolerance
- degraded mode
- stale lease cleanup
- crash recovery
- manifest / receipt guard

### Step 9. optional PTY pilot
- lane 1개만 pilot
- 성능/안정성 비교

## 12. 롤백 전략

전환 중 문제가 생기면 아래 수준으로 롤백합니다.

1. launcher만 기존 렌더링으로 복귀하지 않고, **이전 stable status reader**로만 복귀합니다.  
2. supervisor takeover 이후 문제가 생기면 supervisor를 끄고 watcher-exporter writer를 복원합니다.  
3. TmuxAdapter는 유지하되 wrapper event 사용만 비활성화할 수 있습니다.  
4. receipt gate에 문제 있으면 round close를 막고 manual operator review로 전환합니다.

단, run-scoped layout 도입 이후에는 기존 전역 state 삭제 방식으로 되돌리지 않는 편이 안전합니다.

## 13. 미해결 이슈

아래는 이번 개정판에서도 남겨둘 수 있는 open question입니다.

1. controller polling만 유지할지, 이후 SSE/WebSocket을 추가할지
2. lane별 retry 횟수와 타임아웃 숫자를 환경별로 얼마나 조정할지

## 14. 결론

목표 아키텍처의 핵심은 substrate 교체가 아니라 **책임 분리**입니다.  
특히 compatibility 범위, control block, single-writer takeover, wrapper event 경계를 고정하지 않으면 PTY로 옮겨도 현재 문제가 다른 모습으로 반복될 가능성이 큽니다.
