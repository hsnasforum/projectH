# projectH Pipeline Runtime QA 시험계획서
버전: v1.0  
작성일: 2026-04-10  
문서 성격: QA / 검증 계획

## 1. QA 목표

이번 QA의 목표는 “기능이 돌아간다”를 넘어서, 다음을 증명하는 것입니다.

1. 24시간 무인 운용이 **채택(adoption) / baseline gate 기준**으로 가능한가
2. 장애가 나도 복구 또는 안전정지가 일관되게 가능한가
3. 상태 불일치가 구조적으로 줄었는가
4. launcher가 더 이상 상태 추론의 원인이 아닌가
5. status v1이 current operator workflow를 충분히 대체하는가

> 참고: 1차 자동화 milestone은 **PR 전 정지 + 6시간 synthetic soak 통과**입니다.
> 24시간 무인 운용 기준은 이후 채택/adoption gate로 올립니다.
> 채택 이후의 day-to-day 기본 검증 authority는 `05_운영_RUNBOOK.md §3.5 현재 검증 원칙`이며,
> `launcher live stability gate + incident replay + 실제 작업 세션` 3축으로 정의됩니다.
> 이 문서의 soak test 계획은 baseline evidence + adoption/cutover gate 목적이고, 기본 검증 메뉴를 이 문서로 대체하지 않습니다.

## 2. 시험 범위

### 포함
- supervisor lifecycle
- launcher start/stop/restart/status
- backend adapter health
- run-scoped state
- control block / active control surface
- receipt 생성/누락/불일치
- recovery / degraded mode
- fault injection
- migration shadow / takeover 검증

### 제외
- `app.web` 사용자 기능 회귀
- 모델 응답 품질 평가
- 장기 memory 기능

## 3. 시험 전략

## 3.1 Unit Test
대상:
- state machine
- receipt validator
- control block calculator
- run_id allocator
- stale lease cleaner
- wrapper event parser
- adapter health parser

## 3.2 Integration Test
대상:
- launcher ↔ supervisor
- supervisor ↔ TmuxAdapter
- supervisor ↔ watcher/orchestrator
- supervisor ↔ wrapper-events
- receipt write/read
- restart recovery
- shadow diff / takeover

## 3.3 End-to-End Test
대상:
- start → ready → dispatch → verify → receipt → close → stop 전체 흐름
- control slot 변경 → status control block 반영 흐름

## 3.4 Soak Test (채택/baseline 전용)
이 항목은 **채택(adoption) / baseline gate** 용도이며, day-to-day 기본 검증은 `05_운영_RUNBOOK §3.5` 3축이 authority입니다.
대상:
- 6시간 synthetic soak (1차 automation baseline)
- 24시간 무인 운용 (채택 baseline gate)
- 장시간 polling / health check / recovery 누적 영향

## 3.5 Fault Injection
대상:
- lane process kill
- backend session loss
- stale pid
- corrupt state
- lock file 유실/중복
- manifest mismatch
- supervisor restart during active round
- wrapper event 중단
- active control slot 교체 타이밍 경쟁

## 4. 환경

### 권장 환경
- WSL/Linux 운영 환경
- tmux backend
- 동일 repo 기준 synthetic workload
- event log 및 status log 수집 활성화
- short/long synthetic soak는 `scripts/pipeline_runtime_gate.py synthetic-soak`로 temp workspace에서 실행할 수 있어야 함
- synthetic soak는 real repo `work/`, `verify/`를 직접 오염시키지 않고 temp workspace에서 artifact/control/receipt를 생성해야 함

### 데이터
- 최소 100개 이상 synthetic artifact
- 길이/크기/수정 주기가 다른 혼합 workload
- verify refutation 발생 케이스 포함
- control slot 전환 케이스 포함

## 5. 핵심 검증 항목

## TC-001. Launcher thin client 검증
절차:
1. launcher 실행
2. status 조회
3. launcher 프로세스의 파일 I/O/서브프로세스 호출 확인

기대 결과:
- launcher가 tmux 직접 호출 없이 동작
- watcher.log 직접 읽기 없음
- `work`, `verify` 전체 재귀 스캔 없음

## TC-002. Migration 단계별 writer ownership 검증
절차:
1. Step 1 환경에서 production `status.json` writer 확인
2. Step 2 shadow mode에서 production/shadow 경로 확인
3. Step 3 takeover 후 writer 재확인

기대 결과:
- Step 1: watcher-exporter 1개만 production writer
- Step 2: supervisor는 shadow writer만 수행
- Step 3: supervisor만 production writer
- 어느 단계에서도 둘이 같은 production 파일을 동시에 쓰지 않음

## TC-003. Control block 일관성 검증
절차:
1. `.pipeline/claude_handoff.md` 활성화
2. `CONTROL_SEQ` 증가 후 다른 slot 활성화
3. status `control` block 확인

기대 결과:
- `active_control_file`이 현재 active slot과 일치
- `active_control_seq`가 최신 valid control과 일치
- stale control 후보가 active처럼 노출되지 않음

## TC-004. Shadow diff 검증
절차:
1. supervisor shadow mode 활성화
2. start → dispatch → verify → stop 흐름 실행
3. production status와 shadow status 비교

기대 결과:
- 핵심 필드(runtime_state, lane state, active round, control block, last receipt)가 일치
- diff 발생 시 이유가 로그에 남음

## TC-005. Start readiness 검증
절차:
1. start 요청
2. lane wrapper `READY` 이전 status 확인

기대 결과:
- 단순 session alive만으로 RUNNING/READY가 되지 않음
- `STARTING` 상태가 유지됨
- wrapper event 도착 후에만 `READY` 전이

## TC-006. TmuxAdapter 경계 검증
절차:
1. backend를 tmux로 실행
2. attach / health / restart_lane 수행

기대 결과:
- 상위 계층은 adapter 인터페이스만 사용
- tmux 명령 상세가 launcher에 없음

## TC-007. Run-scoped state 검증
절차:
1. run A 시작 후 종료
2. run B 시작
3. stale state/lock/manifest 혼입 여부 확인

기대 결과:
- run A 정보가 run B current truth로 보이지 않음

## TC-008. Receipt gating 검증
절차:
1. verify artifact 생성
2. receipt 생성 지연
3. status 확인

기대 결과:
- latest verify는 보이지만 round는 `CLOSED`가 아님
- receipt 기록 후에만 `CLOSED`
- receipt가 `/verify`를 대체하지 않음

## TC-009. Status 일관성 검증
절차:
1. launcher / controller / raw status 파일 비교
2. 동일 시점 상태 비교

기대 결과:
- lane state, active round, degraded reason, control block이 동일

## TC-010. Event completeness 검증
절차:
1. start부터 stop까지 실행
2. events.jsonl 수집

기대 결과:
- 필수 lifecycle event 누락 없음
- seq 역전 없음
- `control_changed` 이벤트 누락 없음
- 멈춤/주의 승격 시 `automation_incident` event와 `status.automation_health` / reason / family / next action이 함께 남음
- `signal_mismatch`, `dispatch_stall`, `completion_stall`, `operator_retriage_no_next_control`, `idle_release_pending`, `lane_recovery_exhausted` family가 raw pane text 없이 구분됨

## TC-010A. Duplicate implement handoff suppression
절차:
1. 같은 handoff SHA를 유지한 `STATUS: implement` control을 다시 active로 둠
2. implement owner pane이 `already completed` 또는 동등한 no-op 문구를 노출하게 함
3. watcher/supervisor/status surface 확인

기대 결과:
- watcher가 verify/handoff-owner triage를 1회만 기록함
- supervisor `events.jsonl`에 `control_duplicate_ignored`가 기록됨
- implement owner lane은 `WORKING`에 고정되지 않고 `READY`로 내려감
- implement owner lane note는 `waiting_next_control`
- 자동 operator stop으로 불필요 승격되지 않음

## TC-010B. Stale operator stop suppression
절차:
1. active control을 `.pipeline/operator_request.md`로 둠
2. operator stop 본문에 blocker `work/...md` 경로를 기록함
3. 동일 경로 artifact를 가진 job state를 모두 `VERIFY_DONE`으로 준비함
4. supervisor/status/events를 확인함

기대 결과:
- runtime `status.control`은 `none`으로 내려감
- compat control slot에는 `operator_request.md`가 debug surface로 남음
- supervisor `events.jsonl`에 `control_operator_stale_ignored`가 기록됨
- stale operator stop 때문에 lane이 `WORKING` 또는 `OPERATOR_WAIT`처럼 잘못 surface되지 않음
- operator gate 후속 처리 중이면 active lane에는 raw note와 별도로 `progress_phase=operator_gate_followup`이 남음

## TC-010C. Stale operator stop auto-followup
절차:
1. `TC-010B` 조건을 만족하는 stale operator stop 상태로 runtime을 시작함
2. startup turn과 watcher raw event를 확인함
3. Codex pane tail 또는 runtime lane status를 확인함

기대 결과:
- watcher startup turn은 `VERIFY_FOLLOWUP`
- watcher raw event에 `operator_request_stale_ignored`와 `verify_control_recovery_notify`가 기록됨
- runtime `status.control`은 계속 `none`
- active verify/handoff-owner lane은 `WORKING`, note `followup`으로 surface될 수 있음
- verify/handoff owner는 stale stop을 그대로 방치하지 않고 다음 canonical control outcome을 다시 작성하려고 시도함

## TC-011. Stop / restart idempotency 검증
절차:
1. stop 2회 연속
2. restart 중복 요청

기대 결과:
- orphan process 없이 안전 처리
- 중복 요청이 중복 실행으로 이어지지 않음

## TC-011A. Duplicate supervisor reconciliation
절차:
1. 같은 `project_root + session`으로 supervisor daemon을 2개 이상 띄움
2. `pipeline_runtime.cli stop` 또는 `restart` 실행
3. `.pipeline/supervisor.pid`, `current_run.json`, live process 목록 확인

기대 결과:
- stop은 matching daemon을 모두 내림
- restart 후에는 live supervisor가 1개만 남음
- `supervisor.pid`는 sole live daemon pid와 일치
- `current_run.json`은 마지막 live run 하나만 가리킴

## TC-012. Lane crash recovery
절차:
1. 작업 중 lane 프로세스 kill
2. status 및 recovery 흐름 관찰

기대 결과:
- lane `DEGRADED`
- recovery event 기록
- 정책에 따라 자동복구 또는 안전정지

## TC-012A. Active lane auth/login failure
절차:
1. active control이 열린 lane pane tail에 `API Error: 401`, `Invalid authentication credentials`, `Please run /login`을 노출시킴
2. status와 events를 확인함
3. 자동 recovery/restart 발생 여부를 확인함

기대 결과:
- 해당 lane은 `BROKEN`
- lane note는 `auth_login_required`
- runtime은 `DEGRADED` 또는 정책상 더 강한 `BROKEN`
- blind restart가 반복되지 않음
- operator는 auth 문제를 해결하기 전까지 false `READY`를 보지 않음

## TC-013. Supervisor crash recovery
절차:
1. active round 도중 supervisor kill
2. supervisor 재기동

기대 결과:
- current_run 복구
- duplicate dispatch 없음
- receipt 중복 기록 없음

## TC-014. Wrapper event 중단 검증
절차:
1. lane wrapper heartbeat 중단
2. vendor CLI는 살아 있게 유지
3. status 확인

기대 결과:
- false READY 금지
- lane 또는 runtime가 `DEGRADED`
- 이유가 명시됨

## TC-015. Stale pid 검증
절차:
1. pid 파일만 남기고 실제 프로세스 종료
2. status 조회

기대 결과:
- false ALIVE 금지
- backend health가 우선

## TC-016. Corrupt state quarantine
절차:
1. state JSON 손상
2. supervisor 재시작

기대 결과:
- 손상 파일 quarantine
- 이벤트 기록
- 전체 runtime가 silent failure로 멈추지 않음

## TC-017. Manifest mismatch
절차:
1. receipt 후보와 manifest hash mismatch 유도
2. status 확인

기대 결과:
- `DEGRADED`
- receipt 확정 금지
- mismatch reason 기록

## TC-018. Stale lease cleanup
절차:
1. lease만 남기고 실제 lane 비활성화
2. cleanup 주기 대기

기대 결과:
- TTL 경과 후 stale lease 정리
- 새 dispatch 가능

## TC-019. Backend session loss
절차:
1. tmux session 강제 종료
2. supervisor 관찰

기대 결과:
- false READY 없음
- runtime `DEGRADED` 또는 `BROKEN`
- 운영자에게 명시적 원인 노출
- session scaffold 재생성은 안정 윈도우 전 반복되지 않음
- 반복 session loss는 `session_recovery_exhausted` event와 `automation_health=needs_operator`로 남음

## TC-020. Verify refutation flow
절차:
1. verify 완료 후 refutation 유도
2. 새 round 열림 확인

기대 결과:
- 이전 round close semantics 유지
- 새 round 상태 전이가 일관적임

## 6. 24시간 Soak Test 설계 (채택/baseline 전용)

## 6.1 목표
장시간 운용 중 누적 상태 오염, orphan process, 상태 불일치, false READY, receipt 누락이 없는지 **채택 baseline 기준**으로 검증합니다.
이 설계는 채택/cutover 주기 또는 supervisor/watcher/tmux adapter/wrapper event/state writer 계약이 크게 바뀐 경우에 사용하고, day-to-day 기본 검증으로 반복 수행하지 않습니다.

## 6.2 시나리오
- synthetic artifact 지속 생성
- 주기적 dispatch/verify
- 랜덤 lane crash 2~3회
- supervisor restart 1~2회
- stale pid / stale lease / corrupt state injection 각 1회 이상
- active control 교체 2회 이상
- attach/debug 1회 포함 가능

권장 실행 경로 (채택/baseline gate 용도):
- short smoke: `python3 scripts/pipeline_runtime_gate.py --mode experimental synthetic-soak --duration-sec 30 --sample-interval-sec 1 --min-receipts 1`
- adoption gate: 같은 명령 경로로 6h mini soak, 24h soak를 각각 1회 실행
- day-to-day 기본 검증은 위 명령이 아니라 `05_운영_RUNBOOK §3.5` 3축을 사용합니다.

## 6.3 수집 지표
| 지표 | 목표 |
|---|---:|
| total rounds | 충분한 workload 확보 |
| successful closes | 99.5% 이상 |
| duplicate dispatch | 0 |
| receipt missing | 0 |
| orphan process | 0 |
| control mismatch | 0 |
| degraded enter count | 기록 가능 |
| degraded unrecovered | 0 또는 명시적 BROKEN |
| manual intervention | 0 |

## 6.4 실패 기준
아래 중 하나라도 발생하면 soak fail입니다.

- manual intervention 필요
- duplicate dispatch 발생
- false READY / false CLOSED 발생
- receipt 누락
- run state ambiguity
- control block mismatch
- orphan process 미정리
- silent stall 5분 이상

## 7. Internal tooling adoption gate

다음은 **채택(adoption) / cutover 주기의 최소 게이트**입니다. 일상 merge 기본 검증은 `05_운영_RUNBOOK §3.5` 3축으로 수행하며, 아래 게이트는 runtime 계약이 크게 바뀌거나 채택 sign-off가 필요한 경우에 실행합니다.

1. unit test 전부 pass
2. integration smoke pass
3. start/stop/restart e2e pass
4. fault injection subset pass (`fault-check`, synthetic workspace)
5. shadow diff pass
6. 6시간 mini soak pass (채택 전 baseline)

정식 **internal tooling adoption gate**는 추가로 24시간 soak pass입니다.  
이 표현은 제품 본체 release gate가 아니라 pipeline runtime 채택 게이트를 뜻하며, day-to-day 기본 검증 메뉴가 아닙니다.

## 8. 결함 심각도 기준

아래 기준은 채택/cutover QA round 1회를 닫는 조건이며, 이후 day-to-day 기본 검증은 `05_운영_RUNBOOK §3.5` 3축으로 유지합니다.

1. Sev-1 = 0

6. runbook이 실제 장애 시나리오와 맞아야 함

## 11. 결론

이번 QA는 “동작 확인”보다 “운영 신뢰도 증명”에 더 가깝습니다.  
특히 24시간 soak과 fault injection, control block 일관성, shadow-to-takeover 검증을 통과하지 못하면 canonical runtime으로 **채택(adoption)** 하면 안 됩니다.
다만 이 QA gate는 채택/cutover 주기용이며, 채택 이후의 상시 검증은 `05_운영_RUNBOOK §3.5`의 3축(launcher live stability gate + incident replay + 실제 작업 세션)이 기본 경로입니다.
