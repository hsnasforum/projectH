# projectH Pipeline Runtime 마이그레이션 체크리스트
버전: v1.0  
작성일: 2026-04-10  
문서 성격: cutover / rollback 체크리스트  
대상: launcher/runtime 구조 전환

## 1. 목적

본 문서는 기존 tmux+watcher 운영 구조에서 supervisor 중심 구조로 전환할 때,
cutover 순서, legacy shim 제거 순서, rollback decision point를 체크리스트 형태로 정리한 것입니다.

## 2. 전환 원칙

- 한 번에 전체 교체하지 않습니다.
- production status writer는 항상 1개만 허용합니다.
- launcher를 먼저 얇게 만들고, backend 교체는 나중에 합니다.
- `TmuxAdapter`를 먼저 안정화한 뒤 PTY pilot은 선택적으로 진행합니다.
- rollback 가능 상태를 유지한 채 단계적으로 전환합니다.
- export-only는 legacy status surface로만 제한하고, rolling control slot은 제외합니다.

## 3. 사전 준비

- [ ] baseline 24h 로그 또는 최소 6h mini soak 결과 확보
- [ ] 현재 active control slot 동작 확인
- [ ] synthetic workload 준비
- [ ] rollback 담당자 지정
- [ ] `05_운영_RUNBOOK.md` 최신화
- [ ] 본 체크리스트와 QA 계획서 version 일치 확인

## 4. 단계별 cutover

## Stage A — status schema v1 고정
목표: 새 status surface를 먼저 고정합니다.

### 실행 항목
- [ ] `status.json` v1 확정
- [ ] `control` block 포함
- [ ] launcher/GUI reader를 status v1 기준으로 수정
- [ ] raw tmux/log/file scan UI 경로 비활성화 또는 숨김
- [ ] watcher-exporter를 production status writer 1개로 고정

### 확인 항목
- [ ] `active_control_file` 표시
- [ ] `active_control_seq` 표시
- [ ] launcher가 direct `capture-pane`를 호출하지 않음
- [ ] `watcher.log`를 상태 판정에 직접 쓰지 않음

### rollback point
- 문제가 생기면 기존 launcher UI로 돌아가지 말고, **직전 stable status reader**로만 복귀합니다.

## Stage B — supervisor shadow mode
목표: supervisor를 투입하되 아직 production writer로 전환하지 않습니다.

### 실행 항목
- [ ] supervisor skeleton 실행
- [ ] shadow status 출력 경로 분리
- [ ] command API start/stop/restart/attach 구현
- [ ] production writer는 watcher-exporter 유지

### 확인 항목
- [ ] production `status.json` writer = watcher-exporter 1개
- [ ] shadow output writer = supervisor
- [ ] 핵심 필드 diff 수집 가능
- [ ] shadow diff가 허용 범위 안에 있음

### rollback point
- shadow diff가 커지면 supervisor shadow만 끄고 Stage A 상태로 복귀합니다.

## Stage C — supervisor takeover
목표: supervisor를 status/control current truth writer로 승격합니다. receipt writer 도입은 Stage E 범위입니다.

### 실행 항목
- [ ] supervisor를 production `status.json` writer로 전환
- [ ] watcher-exporter 비활성화
- [ ] control 계산 로직을 supervisor 기준으로 고정
- [ ] launcher 명령 경로를 supervisor로 고정

### 확인 항목
- [ ] production `status.json` writer = supervisor 1개
- [ ] watcher-exporter off 확인
- [ ] control block / active round / lane state 일관성 확인
- [ ] duplicate status/control writer 없음
- [ ] receipt ownership 전환은 아직 수행하지 않음 (Stage E scope)

### rollback point
- 불일치가 생기면 watcher-exporter를 재활성화하고 supervisor production writer를 끕니다.
- 단, takeover 이후에도 schema v1 자체는 유지합니다.

## Stage D — TmuxAdapter 분리
목표: tmux 직접 호출을 backend detail로 격하합니다.

### 실행 항목
- [ ] launcher의 tmux 직접 호출 제거
- [ ] `TmuxAdapter`에 spawn/attach/health 집중
- [ ] shell script 의존 최소화
- [ ] backend metadata 경로 분리

### 확인 항목
- [ ] launcher 코드에서 tmux 명령 문자열 제거
- [ ] attach는 supervisor → adapter 경로로만 실행
- [ ] health는 adapter가 계산하고 status는 supervisor가 surface

### rollback point
- attach나 health가 깨지면 supervisor는 유지하되 adapter 경계 이전 stable commit으로 되돌립니다.

## Stage E — run-scoped state + receipt
목표: stale state와 close ambiguity를 제거합니다.

### 실행 항목
- [ ] `.pipeline/runs/<run_id>/...` 구조 도입
- [ ] `current_run.json` 도입
- [ ] receipt writer 도입
- [ ] receipt current truth writer를 supervisor로 확정
- [ ] 전역 state/lock/manifests 초기화 제거
- [ ] latest-work/latest-verify는 compat read model로만 유지

### 확인 항목
- [ ] receipt current truth writer = supervisor 1개
- [ ] run A와 run B 상태 혼입 없음
- [ ] receipt 없는 close 금지
- [ ] `/work`, `/verify`가 여전히 persistent truth로 유지
- [ ] compat surface가 control plane으로 오용되지 않음

### rollback point
- receipt gate 문제 시 close를 막고 operator review로 전환합니다.
- run-scoped layout 이후에는 전역 초기화 방식으로 되돌리지 않습니다.

## Stage F — wrapper contract + fault tolerance
목표: readiness와 recovery를 공식 이벤트로 옮깁니다.

### 실행 항목
- [ ] lane wrapper 배치
- [ ] `wrapper-events/<lane>.jsonl` 수집
- [ ] 필수 이벤트 구현
- [ ] parser는 wrapper 내부로만 제한
- [ ] stale lease / crash recovery / degraded mode 활성화

### 확인 항목
- [ ] STARTING 유지가 wrapper `READY` 기준으로 작동
- [ ] wrapper heartbeat 중단 시 false READY 없음
- [ ] lane crash recovery 동작
- [ ] orphan process 없음

### rollback point
- wrapper event가 불안정하면 TmuxAdapter는 유지하고 wrapper event 사용만 끌 수 있습니다.
- 단, launcher가 raw pane text를 다시 읽는 방식으로 복귀하면 안 됩니다.

## Stage G — internal tooling adoption gate
목표: canonical runtime 채택 여부를 결정합니다.

### 실행 항목
- [ ] short synthetic smoke pass
- [ ] shadow diff pass
- [ ] fault injection pass (`fault-check`, synthetic workspace)
- [ ] 6h mini soak pass
- [ ] 24h soak pass
- [ ] runbook / migration checklist / QA 결과 동기화

### 채택 기준
- [ ] manual intervention 0
- [ ] duplicate dispatch 0
- [ ] receipt missing 0
- [ ] control mismatch 0
- [ ] false READY / false CLOSED 0

### rollback point
- 게이트 실패 시 canonical runtime 채택을 보류하고, 마지막 stable stage로 후퇴합니다.

## 5. legacy shim 제거 순서

아래 순서보다 앞당기지 않습니다.

1. launcher direct tmux/log/file scan 제거
2. watcher-exporter shadow-only 또는 off 전환
3. legacy status snapshot 축소
4. legacy compat helper 정리
5. optional PTY pilot 이후 tmux 제거 여부 검토

## 6. 금지 사항

- [ ] production `status.json`에 동시 2 writer 허용
- [ ] `.pipeline/*.md` control slot을 export-only compat로 오해
- [ ] `/work`, `/verify`를 receipt로 대체
- [ ] wrapper 이전 raw pane parser를 launcher/GUI에 재도입
- [ ] run-scoped layout 이후 전역 state 삭제 방식 복귀

## 7. cutover 종료 선언 조건

- [ ] supervisor 단일 writer
- [ ] status v1 + control block 고정
- [ ] receipt gate 활성화
- [ ] wrapper readiness 활성화
- [ ] 24h soak pass
- [ ] 운영 문서와 실제 절차 일치

## 8. 한 줄 결론

이번 cutover의 핵심은 backend 교체가 아니라 **writer ownership, control surface, receipt semantics, wrapper readiness를 단계별로 굳히는 것**입니다.
