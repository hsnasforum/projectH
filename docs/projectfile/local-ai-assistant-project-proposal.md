# 로컬 AI 비서 프로젝트 제안서 및 실행 지침

- **영문 제목:** Local AI Assistant Project Proposal & Execution Guide
- **버전:** v1.1
- **작성일:** 2026-03-25

## 1. 문서 목적과 적용 범위
- 본 문서는 기존 초안·제안서를 하나의 실행 문서로 통합하고, 상업화·운영·안전성 관점에서 부족했던 부분을 보완한 정리본입니다.
- 기준 가정은 로컬 우선(Local-first), 승인 기반 실행, 공급자 중립 구조, 자체 브랜드 우선, 상업화 가능한 권리 구조입니다.
- 즉시 목표는 거대 모델 자체 개발이 아니라, 문서 읽기·요약·검색·승인 후 저장이 가능한 로컬 AI 비서 MVP를 안전하게 출시 가능한 구조로 만드는 것입니다.

## 2. 이번 개정에서 보완한 핵심 사항
| 초안 기준 부족점 | 이번 보완 내용 | 기대 효과 |
| --- | --- | --- |
| 초안과 운영 규칙이 분리되지 않음 | 제품 제안서와 에이전트 운영 지침을 구분해 역할을 명확히 정리 | 기획·구현·검토 책임이 섞이지 않고 유지보수가 쉬워집니다. |
| 특정 런타임 종속 가능성 | Codex 기본 설정은 중립형으로 두고, Ollama는 선택형 설정으로 분리 | 로컬 런타임을 붙이되 제품 구조는 공급자 중립으로 유지할 수 있습니다. |
| 상업화 검토 기준이 느슨함 | 코드·모델·데이터·자산·상표를 분리한 권리 점검 원칙을 추가 | 상용화 시 법적 리스크를 초기에 차단할 수 있습니다. |
| 보안 정책이 선언적 수준에 머무름 | 승인 기반 쓰기, 로그 기록, 파괴적 동작 금지, 허용 경로 제한 원칙을 명문화 | 로컬 실행 도구의 신뢰성을 제품 수준으로 끌어올릴 수 있습니다. |
| 개발 단계 목표가 다소 추상적임 | 수직 슬라이스, 마일스톤, 수용 기준, 즉시 작업 순서를 구체화 | Codex/Claude를 활용한 실제 개발 티켓으로 바로 연결할 수 있습니다. |

## 3. 프로젝트 요약
> 내 PC에서 실행되며, 문서를 읽고 요약하고 검색하고, 승인 기반으로 제한된 작업만 수행하는 로컬 AI 비서 MVP를 만든다.

### 핵심 목표
- 로컬 파일 기반 생산성 도구를 안전하게 제공한다.
- 향후 독자 모델·독자 데이터로 확장 가능한 구조를 선행 확보한다.
- 상업화 시 권리 구조와 브랜드 리스크를 줄인다.

## 4. 제품 원칙 및 의사결정 기준
| 원칙 | 설명 |
| --- | --- |
| 로컬 우선 | 기본 동작은 사용자 장비 안에서 해결하며, 클라우드 의존 기능은 명시적 선택일 때만 도입합니다. |
| 승인 기반 실행 | 읽기 작업을 기본으로 하고, 쓰기·이동·삭제·셸 실행·외부 네트워크는 승인 전제로 취급합니다. |
| 공급자 중립 구조 | 모델, 런타임, 저장소는 교체 가능한 인터페이스로 설계하고 특정 공급자에 하드코딩하지 않습니다. |
| 작은 범위 우선 | MVP 범위를 넘는 기능 추가, 대규모 리팩터링, 과도한 추상화는 기본적으로 보류합니다. |
| 상업화 가능성 확보 | 코드 라이선스, 모델 라이선스, 데이터/자산, 상표를 분리 관리하여 향후 제품화 경로를 막지 않도록 합니다. |
| 감사 가능성 | 위험 동작은 로그와 승인 단계를 남기고, 숨은 파괴적 동작을 허용하지 않습니다. |

## 5. MVP 범위와 제외 범위
### MVP 범위
- 채팅 기반 요청 입력과 응답 출력
- 로컬 파일 읽기(.txt, .md 우선, PDF는 전략 수립 후 단계적 도입)
- 단일 문서 요약 및 다중 문서 기초 요약
- 파일명 검색 및 텍스트 기반 내용 검색
- 세션 저장과 작업 로그 저장
- 승인 후 새 노트/요약본 저장

### MVP 제외 범위
- 음성 웨이크워드와 상시 대기형 음성 비서
- 모바일 앱 및 다중 메신저 연동
- 브라우저 자동 로그인과 자율 실행형 파괴 작업
- 항상 켜져 있는 백그라운드 에이전트
- 대규모 독자 사전학습을 MVP의 선행 조건으로 두는 것

## 8. 시스템 아키텍처
| 레이어 | 책임 |
| --- | --- |
| App Layer | 사용자 진입점, 채팅/CLI/UI, 승인 프롬프트, 결과 표시 |
| Core Layer | 에이전트 루프, 오케스트레이션, 도구 선택, 승인 게이트 |
| Tools Layer | 파일 읽기, 파일 검색, 요약 저장, 명시적 로컬 작업 도구 |
| Storage Layer | 세션 저장, 작업 로그, 가벼운 사용자 메모리, 로컬 메타데이터 |
| Model Adapter Layer | 공급자 중립 인터페이스, mock/로컬/클라우드 제공자 분리 |

### 첫 수직 슬라이스
1. 로컬 텍스트 또는 마크다운 파일을 읽는다.
2. 모델 어댑터를 통해 요약을 생성한다.
3. 결과를 화면에 보여준다.
4. 사용자가 승인하면 새 마크다운 노트로 저장한다.
5. 모든 주요 행동을 세션 및 작업 로그에 기록한다.

## 9. 보안·승인·로그 정책
| 구분 | 원칙 | 설명 |
| --- | --- | --- |
| 기본 모드 | 읽기 중심 | 삭제·덮어쓰기·이동·셸 실행은 기본 자동화 대상이 아닙니다. |
| 승인 정책 | 명시적 승인 필요 | 새 파일 생성도 승인 후 수행하는 것을 기본으로 둡니다. |
| 허용 범위 | 프로젝트 또는 승인된 작업 디렉터리 우선 | 승인되지 않은 경로 밖 변경을 피하고 차단 규칙을 둡니다. |
| 로그 | 사용자 가시적 로그 보존 | 무슨 작업을, 언제, 어떤 승인 하에 수행했는지 남깁니다. |
| 네트워크 | 기본 비활성 또는 제한 | 클라우드 호출은 명시적 선택 사항으로 취급합니다. |

## 11. 단계별 로드맵
| 단계 | 목표 | 주요 산출물 |
| --- | --- | --- |
| Phase 0 | 프로젝트 제어층 정리 | AGENTS.md, config, skills, subagents, 아키텍처/수용 기준 문서 고정 |
| Phase 1 | 첫 수직 슬라이스 구현 | 파일 읽기 → 요약 → 승인 후 저장 → 세션/로그 기록 |
| Phase 2 | 검색과 다중 문서 처리 | 파일명/내용 검색, 다중 문서 요약, 가벼운 메모리 추가 |
| Phase 3 | UX 및 패키징 | 앱 셸 개선, 설치/실행 문서, 라이선스/노티스 정리 |
| Phase 4 | 런타임 확장 | Ollama 등 선택형 로컬 런타임 연결, 제공자 추상화 강화, 평가셋 정비 |
| Phase 5 | 제품 경화(Hardening) | 오류 처리, 승인 UX, 로그 가독성, 초기 상업 패키징 검토 강화 |

## 12. 수용 기준 및 리뷰 게이트
| 구분 | 완료 기준 |
| --- | --- |
| 제어층 | 지침·설정·서브에이전트·스킬이 현재 방향과 일치하고, 기본 설정이 안전하게 동작합니다. |
| 첫 수직 슬라이스 | 로컬 파일 읽기, 요약 생성, 승인 후 저장, 세션·로그 기록이 한 흐름으로 작동합니다. |
| 검색 | 파일명 검색과 텍스트 기반 내용 검색이 작동하고 결과가 추적 가능합니다. |
| 품질 게이트 | 범위 확대, 숨은 파괴 동작, 검증 허위 주장, 공급자 하드코딩이 없어야 합니다. |

## 13. 즉시 실행 권고 사항
1. 문서 기반 의사결정 고정: AGENTS.md/CLAUDE.md/기본 config를 저장소 기준으로 확정합니다.
2. tools/file_reader.py를 .txt/.md 중심으로 정리하고 PDF 전략은 별도 메모로 분리합니다.
3. core/에 승인 흐름 객체를 도입해 단순 boolean 체크를 대체합니다.
4. model_adapter에 첫 실제 제공자를 연결하되 인터페이스는 공급자 중립으로 유지합니다.
5. 수동 검증 체크리스트를 만들어 승인 전 저장·로그 무결성·오류 처리 흐름을 반복 확인합니다.

## 결론
이번 보완안의 핵심은 **기획 문서, 운영 지침, 설정 파일, 구현 마일스톤을 하나의 의사결정 체계로 묶었다는 점**입니다. 즉, 지금 단계에서는 기능을 더 늘리기보다 제어층과 첫 수직 슬라이스를 안정적으로 굳히는 것이 우선입니다.
