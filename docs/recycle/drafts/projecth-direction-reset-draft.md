# projectH 방향성 재정의 초안

- 작성일: 2026-03-26
- 문서 상태: Draft
- 대상: projectH 제품 방향, 웹 검색 범위, 독자모델 장기 전략 정렬

## 1. 한 줄 정의

> projectH는 **로컬 퍼스트 문서 비서**를 중심으로 시작하고, 필요할 때만 **권한 기반 웹 조사 모드**를 통해 외부 근거를 수집하는 개인용 AI 어시스턴트입니다.

## 2. 이 문서를 쓰는 이유

최근 구현 과정에서 웹 검색 품질 문제를 해결하려다 제품이 `로컬 문서 비서`에서 `범용 웹 지식 답변기` 쪽으로 흔들리는 조짐이 있었습니다.  
장기적으로 독자모델을 목표로 하더라도, 현재 단계에서 가장 중요한 것은 모델 소유권이 아니라 **우리 제품이 어떤 질문을 어떻게 풀어야 하는지 정의하는 것**입니다.

이 문서는 아래 질문에 답하기 위해 작성합니다.

1. projectH의 현재 정체성은 무엇인가?
2. 웹 검색은 제품의 핵심인가, 보조 기능인가?
3. 최종 목적이 독자모델일 때 지금 무엇을 해야 하는가?
4. 다음 단계의 구현 우선순위는 무엇인가?

## 3. 현재 상태 진단

### 사실

- 현재 저장소와 동작 흐름의 중심은 **로컬 파일 읽기 / 검색 / 요약 / 승인 후 저장 / 세션 / 로그**입니다.
- 웹 검색은 뒤에 추가된 보조 기능이며, 검색 결과와 원문 발췌를 로컬 JSON으로 저장하고 있습니다.
- 웹 설명형 답변은 최근 `claim`, `slot`, `합의`, `불확실 정보` 구조로 조금씩 보강되고 있으나, 여전히 검색 엔진 결과 품질과 HTML 노이즈에 크게 흔들립니다.
- 최종 답변 품질 이슈는 모델 자체보다도 **검색 품질 / 소스 정책 / 근거 합성 방식 / 답변 템플릿** 문제의 비중이 큽니다.

### 해석

- projectH의 현재 제품 핵심은 **범용 질문 답변**이 아니라 **문서 작업 생산성 + 승인 기반 안전성**입니다.
- 웹 검색은 지금 상태에서 “핵심 제품 기능”이 아니라, **문서 비서를 보조하는 조사 기능**으로 보는 편이 맞습니다.
- 지금 웹 검색이 약한 이유는 모델이 작아서라기보다, 파이프라인이 아직 `검색 요약기`에 가깝고 `조사기`에 가깝지 않기 때문입니다.

## 4. 최종 목적과 현재 단계 목표

### 최종 장기 목표

- projectH에 맞는 **독자모델 또는 자체 튜닝 모델**을 확보합니다.
- 단순한 범용 LLM이 아니라, 아래에 강한 모델을 목표로 합니다.
  - 로컬 문서 읽기
  - 출처 기반 요약
  - 조사형 답변
  - 승인 기반 행동 제안
  - 불확실성 표현

### 현재 단계의 실제 목표

- 독자모델을 바로 만드는 것이 아니라, **독자모델이 풀어야 할 문제를 고정**합니다.
- 아래를 제품 차원에서 먼저 정리합니다.
  - 질문 유형
  - 근거 정책
  - 소스 신뢰도 정책
  - 실패 사례 분류
  - 답변 템플릿
  - 평가셋

### 결론

독자모델은 projectH의 **장기 목적**이 맞습니다.  
하지만 지금 단계의 1차 목표는 **로컬 문서 비서 + 조사형 품질 루프**를 굳히는 것입니다.

## 5. 타깃 사용자

- 주 사용자
  - 긴 문서와 로컬 자료를 자주 읽고 정리하는 개인 사용자
- 보조 사용자
  - 개발자, 연구자, 기획자처럼 로컬 파일 탐색과 요약이 잦은 사용자
- 제외에 가까운 사용자
  - 범용 웹 검색만 빠르게 대체하고 싶은 사용자
  - 최신 뉴스/실시간 웹 답변만 원하는 사용자

## 6. projectH가 풀어야 하는 핵심 문제

1. 로컬 문서를 외부 업로드 없이 읽고 정리하기 어렵다.
2. 파일 저장/변경처럼 위험한 행동은 승인 없이 실행되면 불안하다.
3. 웹 정보가 필요할 때도 출처와 불확실성을 구분해서 보고 싶다.
4. 장기적으로는 이 사용 패턴을 데이터화해 독자모델 학습 목표로 연결하고 싶다.

## 7. 제품 모드 재정의

projectH는 세 가지 모드로 분리해 생각하는 것이 적절합니다.

### 7.1 로컬 문서 비서 모드

가장 중요한 기본 모드입니다.

- 로컬 파일 읽기
- 문서 요약
- 문서 검색
- 메모/요약 저장
- 승인 기반 행동
- 세션/로그 보존

이 모드는 MVP 핵심이며, 제품 정체성의 중심입니다.

### 7.2 웹 조사 모드

보조 모드입니다.

- 사용자가 명시적으로 허용한 경우에만 외부 검색
- 검색 결과는 바로 “정답”이 아니라 **조사 후보군**
- 원문 발췌, 출처 역할, 합의 정도를 함께 표시
- 답변은 `확인된 점 / 불확실 / 출처`를 분리

이 모드는 문서 비서를 보강하는 기능이지, 제품 중심축이 아닙니다.

### 7.3 일반 대화 모드

실험적·보조적 모드입니다.

- small-talk
- 짧은 follow-up
- 문서/조사 흐름 사이의 연결

이 모드는 UX를 부드럽게 하는 역할만 맡고, 정확성 요구가 큰 질문은 문서/조사 모드로 넘겨야 합니다.

## 8. MVP 범위

### 포함

- 채팅 기반 인터페이스
- 로컬 파일 읽기
- 문서 요약
- 문서 검색
- 세션 저장
- 승인 기반 저장
- 작업 로그
- 보조적 웹 조사 모드

### 제외 또는 보류

- 범용 지식 챗봇 경쟁
- 항상 최신 웹 답변을 잘하는 제품으로 포지셔닝
- 대규모 독자 사전학습
- 브라우저 자동 로그인/자동 조작
- 자율 실행형 에이전트

## 9. 웹 검색 기능 재정의

현재 웹 검색은 `검색 -> 상위 결과 일부 읽기 -> 요약` 구조에 가깝습니다.  
이 방식은 홍보 문구, 이벤트 기사, 포털 메뉴, 푸터 노이즈에 쉽게 끌립니다.

앞으로는 웹 검색을 아래처럼 정의합니다.

### 원칙

- 웹 검색은 **조사 기능**이다.
- 검색 결과는 즉시 답이 아니다.
- 답변보다 먼저 근거 수집과 정제가 우선이다.

### 조사 파이프라인 목표

1. 검색
2. 후보 출처 필터링
3. 원문 스캔
4. claim 추출
5. 핵심 슬롯별 합의 계산
6. 부족 슬롯 2차 검색
7. 합의된 사실만 최종 답변

### 설명형 질의 답변 형식

웹 설명형 답변은 자유 요약문보다 아래 형식을 기본으로 둡니다.

- 한 줄 정의
- 사실 카드
  - 개발
  - 서비스/배급
  - 장르/성격
  - 상태
  - 이용 형태
- 불확실 정보
- 추가 확인 필요
- 근거 출처

## 10. 소스 신뢰도 정책

UI와 내부 정책 모두 아래 구분을 그대로 사용합니다.

1. `official`
2. `news`
3. `database`
4. `wiki`
5. `community`

### 규칙

- 공식 발표/공식 사이트를 가장 우선합니다.
- 데이터베이스/백과는 설명형 질의에 강한 보조 출처로 사용합니다.
- 위키는 `백과`로 뭉뚱그리지 않고 `wiki`로 정직하게 표시합니다.
- 커뮤니티/블로그/포털/영상은 설명형 답변의 주근거가 아니라 fallback에 가깝게 둡니다.
- 최신 발표/신작/출시 상태는 공식 확인이 없으면 비워두는 것이 정직한 답입니다.

## 11. 왜 “독자 품질 루프”가 먼저인가

독자모델은 결과입니다. 먼저 필요한 것은 아래입니다.

### 먼저 필요한 것

- 질문 유형 체계
- 실패 사례 수집
- 정답/오답 기준
- 소스 신뢰도 정책
- 근거 합성 방식
- 회귀 테스트 셋

### 나중에 가능한 것

- 작은 모델/큰 모델 역할 분리
- 자체 튜닝 모델(LoRA/SFT)
- 최종 독자모델 전략

즉 지금은 `모델부터`가 아니라, **모델 학습 타깃을 만드는 단계**입니다.

## 12. 리스크

1. 웹 검색을 계속 확장하다가 제품이 범용 웹 지식 챗봇처럼 보일 수 있습니다.
2. 최신성 민감 질의와 안정적 엔티티 질의를 같은 파이프라인으로 처리하면 품질이 흔들립니다.
3. claim 추출과 합의 계산이 heuristic 기반이라 아직 오탐/누락이 있습니다.
4. `위키 / 공식 / 뉴스 / 커뮤니티` 역할 구분이 UI에 충분히 드러나지 않으면 사용자가 신뢰도를 오해할 수 있습니다.
5. 독자모델을 너무 빨리 목표로 잡으면 제품 기준과 평가 기준이 흐려집니다.

## 13. 단계별 마일스톤

### Milestone 1. 정체성 고정

- 제품 중심축을 `로컬 문서 비서`로 명시
- 웹 검색은 `조사 모드`로 하향 정의
- 질의 분류를 `stable entity / latest topic / ambiguous / opinion`로 정리

### Milestone 2. 조사형 웹 파이프라인

- claim 추출
- 슬롯별 합의 계산
- 부족 슬롯 2차 검색
- 합의 사실 중심 fact card 렌더링

### Milestone 3. 평가 루프

- 실패 사례를 `jsonl`로 저장
- 실패 유형 분류
- 회귀 테스트 세트 축적

### Milestone 4. 역할 분리형 모델 운영

- 작은 모델: 분류/추출
- 큰 모델: 최종 정리
- 또는 웹 조사형 답변에서 자유서술 최소화

### Milestone 5. 자체 튜닝 / 독자모델 준비

- 질문-답변-근거-피드백 데이터셋 정리
- 500~2000쌍 수준의 1차 튜닝 셋 구축
- LoRA/SFT 실험

## 14. 수용 기준

### 제품 수용 기준

- 로컬 문서 작업 루프가 웹 검색 없이도 충분히 유용해야 합니다.
- 웹 검색은 보조 기능으로 붙고, 꺼져 있어도 제품 정체성이 무너지지 않아야 합니다.

### 웹 조사 수용 기준

- 설명형 질의는 `한 줄 정의 + 사실 카드 + 불확실 정보 + 근거 출처` 구조를 기본으로 가집니다.
- 공식 확인 없는 출시 상태/플랫폼/서비스 정보는 억지로 채우지 않습니다.
- 커뮤니티/포털/이벤트 기사 문장이 본문 핵심 정보로 올라오지 않아야 합니다.

### 장기 전략 수용 기준

- 실패 사례와 피드백 로그가 모델 교체/튜닝 이전에 충분히 쌓여야 합니다.
- 독자모델 논의는 평가셋과 품질 루프가 생긴 뒤에 시작합니다.

## 15. 이번 주 바로 손댈 5개

1. `stable entity`와 `latest topic` 질의 분류를 분리합니다.
2. 소스 라벨을 `official / news / database / wiki / community`로 교체합니다.
3. 최신성 민감 질의 전용 답변 템플릿을 만듭니다.
4. 플랫폼·출시일·상태 같은 슬롯은 공식 확인 없으면 미기재로 막습니다.
5. 실패 사례를 `jsonl`로 쌓고, 프롬프트 수정 전에 회귀 테스트부터 붙입니다.

## 16. 결론

projectH의 최종 목적은 독자모델이 맞습니다.  
하지만 현재 단계의 1차 목표는 **로컬 퍼스트 문서 비서와 조사형 품질 루프를 확립하는 것**입니다.

따라서 방향은 아래처럼 정리합니다.

- projectH는 `로컬 문서 비서`를 제품 중심축으로 유지합니다.
- 공개 웹 지식 답변은 `조사 모드`로 분리합니다.
- 우선순위는 `검색 품질 -> 근거 정책 -> 답변 템플릿 -> 평가셋 -> 모델 교체 -> 미세조정/독자모델` 순서로 둡니다.

이 방향이 맞아야, 나중에 만드는 독자모델도 단순한 범용 LLM이 아니라 **projectH에 실제로 필요한 일을 잘하는 모델**이 됩니다.
