# 2026-03-26 Teachability-First Local Agent Roadmap

## 목적

이 문서는 `projectH`의 현재 구현과 장기 북극성을 분리해서 정리한 4단계 로드맵입니다.

- 현재 shipped contract: 로컬 퍼스트 문서 비서 웹 MVP
- 장기 북극성: 사용자가 하나씩 가르치고, 나중에는 실제 프로그램까지 조작할 수 있는 학습 준비형 로컬 개인 에이전트

현재 단계는 모델 자체 학습이나 일반 프로그램 조작이 아니라, 그 방향으로 갈 수 있는 제품 루프와 데이터 구조를 먼저 고정하는 단계입니다.

## 현재 위치

현재 구현은 아래에 해당합니다.

- 로컬 문서 읽기
- 문서 요약 / 문서 검색 / 일반 채팅
- 승인 기반 저장과 reissue
- 근거 / 출처 / 요약 반영 구간 표시
- 세션 저장, 웹 조사 기록, task log
- 웹 조사의 보조 모드

즉 현재 `projectH`는 **1단계 초입**에 있습니다.

## 4단계 로드맵 요약

1. 문서 중심의 안전한 로컬 비서
2. 교정과 승인으로 길들여지는 학습 준비형 비서
3. 승인 기반의 제한적 로컬 오퍼레이터
4. 개인화된 로컬 에이전트 모델

## 1단계: 문서 중심의 안전한 로컬 비서

### 단계 정의

문서 작업을 좁고 안전한 영역으로 고정하고, 근거·승인·감사 가능성을 제품 계약으로 굳히는 단계입니다.

### 핵심 가치

- 로컬 파일 기반 생산성
- 위험 작업의 승인 게이트
- 저장 가능한 세션, 근거, 로그
- 웹 조사 결과의 보조적 활용

### 현재 구현과 맞닿는 범위

- 파일 요약
- 문서 검색
- 승인 기반 저장
- evidence / source panel
- response origin
- web investigation history

### 이 단계의 종료 기준

- 반복 사용 가능한 핵심 문서 워크플로우 1개 이상이 안정화됨
- 어떤 행동이 승인 대상인지 제품적으로 명확해짐
- 근거와 로그가 사용자/운영자 모두에게 설명 가능해짐
- 테스트와 문서가 현재 계약을 안정적으로 고정함

### 아직 하지 말아야 할 것

- 자동 프로그램 조작
- 모델 자체 학습이라고 홍보하는 것
- 광범위한 범용 챗봇 확장

## 2단계: 교정과 승인으로 길들여지는 학습 준비형 비서

### 단계 정의

사용자의 수정, 승인, 거절, 말투 선호, 형식 선호를 구조적으로 저장하고 다음 답변에 반영하는 단계입니다.

### 핵심 가치

- “가르치면 반영된다”는 체감
- 사용자의 작업 스타일과 선호가 누적됨
- 향후 개인화 모델 학습용 데이터셋의 씨앗 형성

### 필요한 핵심 기능

- 원본 응답과 사용자 수정본의 쌍 저장
- 승인 / 거절 이유 저장
- 선호 규칙 추출과 로컬 저장
- 다음 답변 전 적용되는 preference memory
- 같은 실수 재발 여부를 보는 간단한 eval

### 이 단계에서 쌓이는 자산

- corrected output pairs
- acceptance / rejection traces
- preference rules
- response quality feedback
- 문서 종류별 좋은 예시와 나쁜 예시

### 이 단계의 종료 기준

- 사용자가 “이전보다 내 취향에 맞아졌다”고 느낄 수 있음
- 교정 데이터가 구조화되어 재사용 가능함
- 단기 메모리와 장기 선호 메모리가 분리되어 있음
- 승인 / 피드백 / 교정 로그가 eval 자산으로 승격 가능함

## 3단계: 승인 기반의 제한적 로컬 오퍼레이터

### 단계 정의

문서 작업을 넘어서, 승인된 범위 안에서 특정 프로그램이나 로컬 툴을 제한적으로 조작하는 단계입니다.

### 핵심 가치

- “답만 하는 비서”에서 “행동하는 비서”로 확장
- 사용자의 지시를 실제 작업 완료까지 이어줌
- 행동이 모두 관찰 가능하고 재현 가능함

### 허용되어야 하는 행동 특성

- 관찰 가능
- 승인 가능
- 취소 가능
- 로그로 재현 가능
- 위험도별 경계가 명확함

### 초기 대상 예시

- 로컬 파일 정리
- 지정된 앱이나 웹 표면에서의 반복 작업
- 문서 저장 / 분류 / 후속 정리

### 이 단계의 종료 기준

- 적어도 1개 이상의 좁은 프로그램 조작 루프가 안정적으로 동작
- 각 행동에 대한 approval / audit / rollback 전략이 존재
- 사용자가 “내가 시키는 방식대로 행동한다”는 신뢰를 가질 수 있음

### 이 단계에서 주의할 것

- 범용 데스크톱 자동화로 급격히 넓히지 않기
- 승인 없는 행동 금지
- 실패했을 때 침묵하지 않기

## 4단계: 개인화된 로컬 에이전트 모델

### 단계 정의

앞선 단계에서 모은 교정, 선호, 승인, 행동 로그를 바탕으로 로컬 적응형 모델이나 독자 모델 계층을 운영하는 단계입니다.

### 핵심 가치

- 사용자별로 길들여진 에이전트
- 단순 프롬프트 재주입을 넘어선 지속적 적응
- 장기적으로는 독자 모델 기반 사업의 핵심 자산 형성

### 가능한 기술 형태

- preference ranker
- adapter / LoRA 기반 미세조정
- task-specific policy model
- retrieval + memory + personalization hybrid

### 필수 전제

- 충분한 품질의 교정 데이터
- 승인과 거절 로그의 구조화
- 행동 로그와 결과 로그의 정제
- 회귀 평가 세트
- rollback 가능한 학습 / 배포 운영

### 이 단계의 종료 기준

- 개인화 품질이 측정 가능함
- 재학습 또는 적응 주기가 운영 가능함
- 안전성과 사업성이 동시에 설명 가능함

## 단계별 공통 원칙

- 로컬 퍼스트
- 승인 기반 안전성
- 감사 가능한 행동
- 사실과 근거의 분리
- 현재 구현과 장기 목표의 분리
- 외부 벤더 종속 최소화

## 지금 당장 하지 말아야 할 것

- “이미 학습하는 모델”처럼 포장하기
- 범용 프로그램 조작을 먼저 붙이기
- 승인 구조 없이 행동 자동화를 확장하기
- 장기 북극성을 현재 shipped feature처럼 문서화하기

## 지금 바로 집중할 다음 우선순위 3개

1. 문서 비서의 핵심 반복 워크플로우 1개를 정식 계약으로 고정
2. 사용자 교정 / 승인 / 거절 / 선호 저장 스키마 설계
3. 다음 답변에 선호를 반영하는 preference memory와 그 eval 기준 설계

## 오픈 질문

1. 개인화 데이터의 저장 단위를 세션 기준으로 볼지, 사용자 기준으로 볼지?
2. 선호 메모리와 장기 학습 데이터 승격 기준을 어떻게 나눌지?
3. 프로그램 조작의 첫 대상은 브라우저인지, 로컬 파일 작업인지, 특정 앱인지?
4. 로컬 전용 운영과 향후 사업화 운영 경계를 어디서 나눌지?
